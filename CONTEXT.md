# MTG News — Context

## Description
MTG news & sentiment scanner. Ingests YouTube videos from watched channels, extracts transcriptions, and produces AI-powered summaries and sentiment analysis using a local LLM.

## Glossary

| Term | Definition |
|---|---|
| **Signal** | The primary domain object. A unit of raw text (transcription) extracted from a source video, along with its metadata (video ID, channel, title, published date). The Signal is what the system processes — not the video itself. |
| **Channel** | A YouTube channel on the WatchList. Identified by its YouTube channel ID. The system polls Channels for new videos. |
| **WatchList** | The configured set of Channels the system monitors. Stored persistently in SQLite. |
| **Transcription** | The caption/subtitle text extracted from a YouTube video using `yt-dlp`. May be manually authored or auto-generated by YouTube. |
| **Summary** | An AI-generated concise summary of a Signal's MTG-relevant content, produced by the local LLM. |
| **Sentiment** | An AI-generated sentiment assessment of a Signal. Two levels: **Overall** (single label for the entire Signal) and **Per-Entity** (sentiment labels for each MTG entity mentioned — cards, sets, players). |
| **Entity** | An MTG-relevant concept mentioned in a Signal: a card name, set name, player name, format, etc. Extracted and scored by the LLM during sentiment analysis. |
| **Poll** | A single execution of the scheduled ingestion cycle: fetch RSS feeds for all Channels, identify new videos, extract transcriptions, run LLM analysis, persist results. Runs once daily. |

## Decisions

### Acquisition
- **YouTube only** for now, architecture designed for additional channels later.
- **yt-dlp** extracts caption/subtitle tracks from YouTube. No API key required. If a video has no captions available, it is skipped and logged.
- **YouTube RSS feeds** (`/feeds/videos.xml?channel_id=...`) for discovering new videos from watched Channels. No API key required. Returns latest ~15 videos per channel.

### Processing
- **Local LLM** via LM Studio at `http://127.0.0.1:1234/v1/chat/completions` (OpenAI-compatible endpoint).
- **Model:** `qwen/qwen3.6-27b`.
- Each Signal produces: Summary + Overall Sentiment + Per-Entity Sentiment.

### Persistence
- **SQLite** for all data: WatchList (channel IDs), processed Signals (video IDs, transcriptions, summaries, sentiment), and Entity mentions.
- Deduplication by YouTube video ID — each video is processed at most once.
- **Schema initialization** uses simple `CREATE TABLE IF NOT EXISTS` in a single module called at app startup — no migration runner framework. Greenfield project with stable schema.

### Polling & Background Worker Model
- **Queue-based async polling via SQLite job table.** Trigger enqueues a `PollRun` row → background worker (separate file/module, runs in-process but on its own loop) picks up the job → processes channels one at a time → updates progress in a `poll_run_progress` table. UI polls for status via HTMX `<meta refresh>` or periodic `hx-get`.
- **Scheduled polling** also creates `PollRun` entries — triggers once daily via `node-cron` or equivalent, runs the same worker path as manual trigger (single source of truth).

### Data Model — Transcription Storage
- Captions from yt-dlp include per-segment timing. Store transcription as a **JSON blob in a SQLite TEXT column** (`TEXT NOT NULL`) containing an array of `{text: string, start: number, end: number}` objects. One JSON blob per Signal — sufficient given no ad-hoc segment-level queries are needed. Timestamp anchor links are generated server-side using `start` offsets to produce `[T:ss]` references in summaries that link to `<span id="t-ss">` anchors in the displayed transcription.

### Schedule
- **Once daily** polling interval (triggered by cron-like scheduler, same code path as manual poll trigger).

### Tech Stack
- **Node.js / TypeScript** (backend + server-side code)
- Express (TypeScript, with `ts-node-dev` or `tsx` for dev hot-reload)
- EJS template engine (`@types/ejs`) — minimal logic, `include()` for layout partials
- HTMX for AJAX-driven HTML swaps (pagination, channel filter, poll trigger, detail navigation)
- Alpine.js (CDN in layout) — lightweight reactivity for toggle pills + signal row expand/collapse
- Tailwind CSS via CDN (`<script src="https://cdn.tailwindcss.com">`) for dev; production build optional later
- SQLite via `better-sqlite3` (synchronous, native bindings)
- `yt-dlp` via Node.js child_process or wrapper
- RSS parsing via standard XML parser
- HTTP client for LM Studio OpenAI-compatible API

### Navigation Structure (Three Top-Level Pages)

| Page | Route Pattern | Purpose |
|---|---|---|
| **Signal Viewer** | `/signals` | Paginated signal list + channel filter bar (see below). Clicking a row navigates to that Signal's detail page. |
| **Signal Detail** | `/signals/:id` | Full view of one Signal: key takeaways summary section, then expandable full transcription section with timestamp anchors. Timestamp links in the summary (`[T:ss]`) are standard HTML `<a href="#t-{ss}">` anchors that scroll to corresponding sections in the transcription below on the same page — similar to Wikipedia's internal section links. |
| **Run History** | `/polls` | Paginated table of all ingestion runs, styled like the signal list (date/time/status/new signals columns). Clicking a run row navigates to detail showing per-channel results. |
| **Admin Panel** | `/admin` | WatchList channel management + poll trigger button + quick status widget. |

All four pages share one `layout.ejs` with persistent sidebar nav. Navigation between Signal Viewer, Signal Detail, Run History, and Admin Panel is full page transitions (not HTMX fragment swaps) — only the signal table rows within `/signals` are swapped via HTMX for pagination/filtering to keep the toggle pills static.

#### Channel Filter UI — Toggle Pills
- Channel filter bar above the Signal table uses **toggle pill buttons** styled with Tailwind CSS utility classes: `rounded-full`, active = `bg-blue-600 text-white`, inactive = `bg-gray-200 text-gray-700`.
- Powered by a tiny Alpine.js store (`channelFilterStore`) tracking selected channel IDs. On selection change, HTMX fires a debounced `hx-get` to reload the signal table fragment only — no full page navigation for filtering.
- Same toggle pill approach used for admin-side WatchList management (uniformity across all channel-related UI).

#### Signal List View
- Paginated table/list (**25 items/page**) with columns: published date, time, 50-char truncated summary, sentiment score (numeric 1–5 scale with **red→green** gradient background coloring based on score value — `bg-red-600` for 1 through `bg-green-600` for 5).
- **Sort order:** newest first by processing time (`created_at DESC`).
- **Channel filter bar** above the table: same toggle pill buttons as described above, rendered via EJS partial `_channelPills.ejs`, powered by Alpine.js store. HTMX swap on selection change reloads only the signal table fragment.

#### Signal Detail View (full page at `/signals/:id`)

**Layout:** Single column, centered within the main content area (max-width ~800px). No table — this is a detail reading view.

1. **Header bar:** Signal title (or "No title"), channel pill badge, published date/time.
2. **Key Takeaways section:** Bullet list of AI-generated key takeaways from the summary. Each takeaway may contain inline `[T:ss]` timestamp links embedded within the text.
3. **Full Transcription section:** Expandable/collapsible (initially collapsed). Renders each transcription segment as a paragraph prefixed with its own `[T:ss]` label. Timestamp anchor markers (`<span id="t-{ss}"></span>`) are placed before each transcript paragraph so summary links can jump to them.

**Timestamp linking mechanism:** The summarization prompt instructs the LLM to produce **section-level summaries with `[T:ss]` timestamp references** matching transcription segment `start` seconds (integer). On the detail page, these render as standard HTML anchor tags `<a href="#t-{ss}">[T:XX]</a>` within summary/takeaway text. Clicking one triggers browser scroll to `<span id="t-{ss}" class="anchor"></span>` in the transcription section below — same-page intra-document navigation like Wikipedia article TOC links.

**Timestamp link interaction:** When clicked, the target transcript paragraph briefly highlights (via HTMX or Alpine event) to signal the jump target, then fades back to normal.

#### Run History Page (`/polls`)

**List view:** Paginated table/list (**25 items/page**, same styling language as Signal Viewer), with columns:
| Column | Content |
|---|---|
| Date / Time | When the poll ran (triggered time) |
| Status | Overall status of the run (`running`, `done`, `failed`) — colored badge |
| New Signals | Count of new signals processed this run |
| Channels | Brief summary: "3/5 done, 1 failed" or similar aggregate |

- **Sort order:** newest first (most recent poll on top).
- Clicking a row navigates to `/polls/:id` showing per-channel breakdown.

#### Admin Panel (`/admin`)
- **WatchList management:** Rich per-channel data — name, avatar (fetched from YouTube), last-poll date, active/inactive toggle. Add channels by YouTube channel ID; on add, fetch and store display name + avatar. Remove or deactivate channels without deleting historical Signals.
- **Poll trigger:** Single "Run Poll Now" button at top of Admin page. Trigger enqueues a `PollRun` job → UI shows progress in the sidebar status widget (or inline). No server-sent events — use HTMX `<meta refresh>` or periodic polling interval for updates. The Run History list itself lives on `/polls`.
- **Authentication:** No auth — internal tool, trusted access only. Admin routes guarded by obscurity (URL hidden in nav).

#### Data Model — Sentiment Score
- Overall sentiment is stored numerically on a **1–5 integer scale** for UI display: `1 = Very Negative`, `2 = Negative`, `3 = Neutral`, `4 = Positive`, `5 = Very Positive`. The LLM also returns the original categorical label for reference.

## Anti-Glossary
Do NOT use these terms (use the glossary equivalent instead):
- ~~Video~~ → use **Signal** (the video is the carrier; the Signal is what we process)
- ~~Transcript~~ → use **Transcription**
- ~~Feed~~ → use **Channel** or **RSS feed** (be specific)
- ~~Analysis~~ → use **Summary** or **Sentiment** (be specific)
