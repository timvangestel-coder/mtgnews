# MTG News — Context

## Description
MTG news & sentiment scanner. Ingests YouTube videos from watched channels, extracts transcriptions, and produces AI-powered summaries and sentiment analysis using a local LLM.

## Glossary

| Term | Definition |
|---|---|
| **Signal** | The primary domain object. A unit of raw text (transcription) extracted from a source video, along with its metadata (video ID, channel, title, published date). The Signal is what the system processes — not the video itself. |
| **Processing State** | A `processing_state` field on Signals: `"pending"` (has transcription, LLM has not run), `"irrelevant"` (LLM ran, content doesn't match topic), `"summarized"` (LLM ran, summary/sentiment/entities stored). Replaces the previous two-column model (`processed_at` + `relevance_status`). Abort/stale cleanup deletes only `"pending"` signals — `"irrelevant"` and `"summarized"` always survive. |
| **Channel** | A YouTube channel on the WatchList. Identified by its YouTube channel ID. The system polls Channels for new videos. Each Channel belongs to exactly one Topic (1:N). Channels with no topic assigned (`topic_id IS NULL`) are blocked from polling. |
| **Topic** | A domain category that groups channels sharing a content focus. Defined by four fields: `key` (machine-readable slug, e.g. `mtg`, `ai`), `short_name` (UI display label, e.g. "MTG", "AI"), `filter_text` (the relevance criteria text injected into the LLM prompt), and `summary_prompt` (nullable TEXT — per-topic custom prompt template for the PromptAssembler). Topics are managed independently of Channels via the Admin panel. Deleting a topic sets channel `topic_id` to NULL (force-delete). |
| **WatchList** | The configured set of Channels the system monitors. Stored persistently in SQLite. |
| **Transcription** | The caption/subtitle text extracted from a YouTube video using `yt-dlp`. May be manually authored or auto-generated by YouTube. |
| **Summary** | An AI-generated concise summary of a Signal's topic-relevant content, produced by the local LLM. |
| **Sentiment** | An AI-generated sentiment assessment of a Signal. Two levels: **Overall** (single label for the entire Signal) and **Per-Entity** (sentiment labels for each entity mentioned). Entity topic membership is derived via join chain (`entity_mentions → signals → channels → topics`), not denormalized. |
| **Entity** | A topic-relevant concept mentioned in a Signal: card name, set name, player name, model, company, etc. Extracted and scored by the LLM during sentiment analysis. Entity type vocabulary is free-form (not constrained per-topic). Topic membership derived via join chain through signals→channels→topics. |
| **SignalContext** | A resolved context object returned by `resolveSignalContext(videoId, db)` containing raw data needed for LLM analysis: `transcriptionJson` (raw JSON blob), `topicId`, `filterText`, and `summaryPrompt` (custom template or NULL). The interface carries raw data — no formatting decisions. Created to eliminate ad-hoc SQL queries in `llm.ts`. |
| **PromptAssembler** | A deep module (`prompt-assembler.ts`) that renders LLM prompts from per-Topic XML templates stored in `topics.summary_prompt`. Uses three exported functions: `assemble(context: SignalContext) → string` (main entry point), `formatTranscription(json) → string` (formats raw transcription JSON into `[T:ss] timestamped text`), and `defaultPromptTemplate() → string` (returns the compiled default template). Templates use `{TRANSCRIPTION}` and `{FILTER_TEXT}` placeholders inside XML tags (`<transcription>`, `<filter_text>`). When `summaryPrompt` is NULL, the default template is used. Pure function — no DB access, no side effects. |
| **Ingestion** | The per-candidate pipeline that transforms an RSS candidate into a persisted Signal: caption extraction → segment grouping → DB insert. Produces an `IngestResult` indicating outcome (ingested, duplicate, or no captions). Called by the Poll worker for each new video candidate. |
| **Poll** | A single execution of the scheduled ingestion cycle: fetch RSS feeds for all active Channels with a non-NULL `topic_id`, identify new videos, run Ingestion for each candidate, then run LLM analysis on new signals. Runs once daily. Channels without a topic are skipped. |
| **E2E Test** | End-to-end browser test using Playwright. Spins up the Express server against an in-memory SQLite DB seeded with fixture data. Tests client-side behavior (Alpine.js toggles, HTMX swaps, transcription expand/collapse) that supertest unit tests cannot verify. |
  | **Run Abort** | A user-initiated cancellation of an active PollRun. Fires an in-memory AbortController to cancel in-flight LLM HTTP requests and stop the worker between tasks. Then deletes signals with `processing_state = 'pending'` by poll_run_id — only unanalyzed signals are removed; irrelevant and summarized signals always survive. Always transitions run status to `done-forced` with correct `new_signal_count`. The worker's pool callbacks guard `incrementDone()` with `signal?.aborted` so in-flight tasks that settle after abort cleanup don't inflate progress counters. Aborted tasks on the AbortError path skip `incrementDone()` entirely. Worker waits for full `pool.drain()` before returning (simple, safe — UI reflects abort immediately via DB update). |
  | **Stale Run** | A PollRun stuck at `status='running'` with no `completed_at`, caused by server shutdown mid-execution. Auto-cleaned on startup: signals with `processing_state = 'pending'` deleted by poll_run_id — aligned with abort logic. Run transitions to `done-forced` if processed > 0, deleted entirely if processed == 0. |
  | **Poll Run UI State** | The progress widget in the Admin [Polling] tab shows per-channel counters during an active PollRun. Single streaming model (no phases): each channel step displays one of five states — `fetching` (blue), `X/Y` processing (blue), `Y/Y` done (green), `none` no signals (grey), `failed` (red). Channels are sorted alphabetically by display name (NULLs first) and remain in that order throughout the run. For aborted runs: channels with 0 summaries show `skipped` (grey), partial work shows `X/N` (amber), fully-completed channels stay `N/N` (green). RSS polling is invisible to the UI (stays "fetching"). Failed signals count toward `done`. No global counter banner. See https://github.com/timvangestel-coder/mtgnews/issues/79. |
 | **Route Module** | An Express Router factory (e.g., `createSignalsRouter`) that receives a service layer dependency and mounts HTTP handlers for one domain area. Each module exports a single `Router` instance. Route modules are thin: parse params → call service → render template or redirect. Mounted in `server.ts` via `app.use()`. |
 | **Service Layer** | Domain-specific orchestration modules that sit between route handlers and pure domain functions. Two naming conventions: `-Service` suffix for read-only query services (`SignalQueryService`, `PollQueryService`), `-Manager` suffix for write/CRUD managers (`ChannelManager`, `TopicManager`). The service wraps DB access and coordinates multi-function operations (e.g., query + format + enrich). Services are the primary unit of testing — unit tests target services directly with in-memory SQLite, while route modules get lightweight HTTP smoke tests. |
  | **HTTP Retry Module** | A generic `fetchWithRetry(url, options, retryConfig)` module that handles timeout via AbortController, external abort signal merging, transient error detection (TypeError, ECONNRESET, ECONNREFUSED), and exponential backoff. Consumer-agnostic: the LLM client is the current adapter; other HTTP clients may adopt it later. |
  | **Concurrency Pool** | A deep module (`concurrency-pool.ts`) that enforces a strict upper bound on parallel task execution using a semaphore counter. Interface: `run(fn)` dispatches a task (queues if at limit), `drain()` waits for all dispatched tasks. Guarantees at most `LLM_CONCURRENCY` (default 3) LLM requests are in flight simultaneously. Replaced the original inline `Promise.race`-based pool that allowed burst concurrency exceeding the configured limit. |
  | **Transcription Merge Module** | A pure-function module (`transcription-merge.ts`) containing `mergeOverlappingSegments()` and `groupSegments()`. Input: raw caption segments. Output: grouped `[{time: number (ms), text: string}]`. No I/O, no format detection — purely algorithmic. Tested in isolation via `merge-segments.test.ts`. |

## Decisions

### Acquisition
- **YouTube only** for now, architecture designed for additional channels later.
- **yt-dlp** extracts caption/subtitle tracks from YouTube. No API key required. If a video has no captions available, it is skipped and logged.
- **YouTube RSS feeds** (`/feeds/videos.xml?channel_id=...`) for discovering new videos from watched Channels. No API key required. Returns latest ~15 videos per channel.
- **Shorts exclusion:** YouTube Shorts are filtered out at RSS discovery by detecting `/shorts/` in the entry's link href. Shorts never become candidates and are silently skipped — no counter, no tracking.

### Processing
- **Local LLM** via LM Studio at `http://127.0.0.1:1234/v1/chat/completions` (OpenAI-compatible endpoint).
- **Model:** `qwen/qwen3.6-27b`.
- Each Signal produces: Summary + Overall Sentiment + Per-Entity Sentiment, generated via a **single merged LLM call** returning one JSON response containing all outputs plus a `"relevant"` boolean. Transcription is sent once per signal, not thrice.
- **Relevance filtering:** The LLM receives the channel's Topic `filter_text` as part of the prompt (resolved via `channels.topic_id → topics`) and must first judge whether the content matches. If irrelevant, it returns `"relevant": false`. The system sets `processing_state = 'irrelevant'`, skips storing summary/sentiment/entities. Relevant signals get full analysis stored with `processing_state = 'summarized'`.
- **Generic analyst role:** The LLM prompt uses a generic role ("You are a content analyst") rather than a domain-specific one. Domain scoping is handled entirely through the Topic's `filter_text`. Entity types are free-form (not constrained per-topic).
- **Prompt Template System:** Per-Topic custom prompts stored in `topics.summary_prompt` (TEXT, nullable). Templates use XML placeholder tags (`<transcription>`, `<filter_text>`) that the PromptAssembler injects at render time. When NULL, a compiled default template is used. Templates edited via Admin [Topics] textarea — no code deploy required for prompt iteration. The assembler formats raw transcription JSON into timestamped text before injection. Module boundary: SignalContext resolves data (raw), PromptAssembler renders (pure function of context → string).

### Persistence
- **SQLite** for all data: WatchList (channel IDs), processed Signals (video IDs, transcriptions, summaries, sentiment), and Entity mentions.
- Deduplication by YouTube video ID — each video is processed at most once.
- **Schema initialization** uses simple `CREATE TABLE IF NOT EXISTS` in a single module called at app startup — no migration runner framework. Greenfield project with stable schema.

### Polling & Background Worker Model
- **Queue-based async polling via SQLite job table.** Trigger enqueues a `PollRun` row → background worker (separate file/module, runs in-process but on its own loop) picks up the job → processes channels sequentially (RSS fetch + caption extraction), then runs LLM analysis through a **Concurrency Pool** controlled by `LLM_CONCURRENCY` env var (default 3). The pool uses a semaphore counter to guarantee at most N tasks run simultaneously. Progress tracked in `poll_run_progress` table. UI polls for status via HTMX periodic `hx-get`.
- **Scheduled polling** also creates `PollRun` entries — triggers once daily via `node-cron` or equivalent, runs the same worker path as manual trigger (single source of truth).

### Data Model — Transcription Storage
- Captions from yt-dlp include per-segment timing. Store transcription as a **JSON blob in a SQLite TEXT column** (`TEXT NOT NULL`) containing an array of `{time: number, text: string}` objects. The `time` field is the group start timestamp in milliseconds, rounded to the nearest second. One JSON blob per Signal.
- **YouTube caption deduplication:** YouTube auto-captions produce overlapping "paint-on" segments (contiguous pairs where text is a superset) plus word-level overlap at segment boundaries. `mergeOverlappingSegments()` performs two-phase cleanup: (1) merge contiguous superset segments, (2) trim overlapping leading words at segment boundaries.
- **Transcription grouping:** After merging, raw segments are grouped into ~10-second windows. Segments within the same 10-second window are concatenated into a single group with one `time` value (the rounded start of the first segment in the group). Grouping happens at ingestion time; the grouped result is stored in the DB, raw segments are discarded.
- **Timestamp display:** Transcription is rendered with `[MM:SS]` format, one timestamp per group. Anchor IDs use the group's `time` value in milliseconds (e.g., `id="t-45000"`).
- **LLM summary timestamps:** The LLM receives the grouped transcription with `[T:ss]` timestamp references (one per ~10-second group). Since the LLM sees the same grouped data stored in the DB, its `[T:ss]` references map directly to stored group timestamps. `injectTimestampAnchors()` converts these to `[MM:SS]` display labels and millisecond anchor IDs (e.g., `[T:45]` → `[00:45]` linking to `#t-45000`).

### Schedule
- **Once daily** polling interval (triggered by cron-like scheduler, same code path as manual poll trigger).

### Tech Stack
- **Node.js / TypeScript** (backend + server-side code)
- Express (TypeScript, with `ts-node-dev` or `tsx` for dev hot-reload)
- EJS template engine (`@types/ejs`) — minimal logic, `include()` for layout partials
- HTMX for AJAX-driven HTML swaps (pagination, channel filter, poll trigger, detail navigation)
- Alpine.js (CDN in layout, loaded with `defer`) — lightweight reactivity for toggle pills, signal row expand/collapse, and Admin Panel tab system (`x-data` + `x-if` templates)
- **HTMX + Alpine integration:** Elements inserted by Alpine's `<template x-if>` are not visible to HTMX at page load (Alpine runs after HTMX scans the DOM). Use `x-init="htmx.process($el)"` on the container `<div>` inside each template so HTMX re-scans and registers `hx-*` attributes after Alpine inserts them.
- Tailwind CSS via CDN (`<script src="https://cdn.tailwindcss.com">`) for dev; production build optional later
- SQLite via `better-sqlite3` (synchronous, native bindings)
- `yt-dlp` via Node.js child_process or wrapper
- RSS parsing via standard XML parser
- HTTP client for LM Studio OpenAI-compatible API
- Vitest + supertest for backend unit/integration tests
- Playwright for E2E browser tests (local only, in-memory SQLite fixture per suite)

### E2E Testing Strategy
- **Playwright** for browser-level tests: verifies Alpine.js toggle pills, HTMX fragment swaps, Signal Detail three-state layout transitions, bidirectional timestamp scrolling
- **In-memory SQLite fixture** spun up in test lifecycle hooks -> each suite gets fresh seeded DB + server on random port -> no shared state, no flaky tests
- **Scope** (priority order): 1) Signal Viewer, 2) Signal Detail, 3) Admin Panel, 4) Run History
- **Local only** for now; CI integration possible later
- **DOM-Only Assertions:** E2E tests must verify behavior through observable DOM state (visibility, CSS classes, text content, scroll position) — never through framework component state (`Alpine.$data()`, HTMX internals, etc). Use page object modules (e.g., `tests/e2e/fixtures/signal-detail-page.ts`) that expose behavior-level methods (`setViewState`, `clickSummaryPill`, `getPaneHeights`) rather than raw locator access.

### Navigation Structure (Four Top-Level Pages)

| Page | Route Pattern | Purpose |
|---|---|---|
| **Signal Viewer** | `/signals` | Paginated signal list + hierarchical topic→channel filter bar. Clicking a row navigates to that Signal's detail page. |
| **Signal Detail** | `/signals/:id` | Full view of one Signal with a three-state layout (SummaryOnly → TranscriptOnly → Split 30/70). Timestamp links rendered as pill badges. Bidirectional navigation between Summary and Transcription panes. See detailed spec below. |
| **Run History** | `/polls` | Paginated table of all ingestion runs, styled like the signal list (date/time/status/new signals columns). Clicking a run row navigates to detail showing per-channel results. |
| **Admin Panel** | `/admin` | Tabbed admin interface: [Channels] topic selector + WatchList CRUD, [Topics] Topic CRUD, [Polling] poll trigger + progress widget. |

All four pages share one `layout.ejs` with persistent sidebar nav. Navigation between Signal Viewer, Signal Detail, Run History, and Admin Panel is full page transitions (not HTMX fragment swaps) — only the signal table rows within `/signals` are swapped via HTMX for pagination/filtering to keep the toggle pills static.

#### Topic + Channel Filter UI — Hierarchical Toggle Pills
- **Two-row filter bar** above Signal table: Row 1 = Topic pills (All + per-topic), Row 2 = Channel pills (All + per-channel).
- Topic pills use same toggle pill styling as channel pills. Default state: no topic selected -> show all signals, all channel pills visible.
- When a topic is selected: channel pills filter to only channels in that topic. Signal table filters by both topic and channel.
- Powered by Alpine.js store tracking `selectedTopic` + `selectedChannel`. On change, HTMX fires debounced `hx-get` to reload signal table fragment only.

#### Signal List View
- Paginated table/list (**25 items/page**) with columns: published date, time, 50-char truncated summary, sentiment score (numeric 1–5 scale with **red→green** gradient background coloring based on score value — `bg-red-600` for 1 through `bg-green-600` for 5).
- **Sort order:** newest first by processing time (`created_at DESC`).
- **Hierarchical filter bar** above the table: topic pills + channel pills as described above, powered by Alpine.js store. HTMX swap on selection change reloads only the signal table fragment.

#### Signal Detail View (full page at `/signals/:id`)

**Layout:** Single column, centered within the main content area (max-width ~800px). Three-state view system managed by Alpine.js (`viewState: 'summary' | 'transcript' | 'split'`).

1. **Header bar:** Signal title (or "No title"), channel pill badge, published date/time.
2. **Key Takeaways section:** Bullet list of AI-generated key takeaways. Inline timestamp links rendered as **pill badges** (`bg-indigo-100 text-indigo-700 rounded`). Clicking enters Split mode.
3. **Full Transcription section:** Renders transcription segments grouped by ~10-second windows. Segment timestamps are also clickable pill badges enabling bidirectional navigation.

**Three-state layout:**
| State | Summary height | Transcript height | Trigger |
|---|---|---|---|
| `summary` (default) | 100% visible | hidden | Page load, or "Back to Summary" button |
| `transcript` | hidden | 100% visible | "Show Full Transcription" button |
| `split` | ~30% of container | ~70% of container | Any timestamp link click |

All transitions animate over 1s via CSS. Both sections always rendered in DOM (no `x-show` collapse).

**Timestamp linking mechanism:** `injectTimestampAnchors()` converts LLM's `[T:ss]` to pill-styled `<a href="#t-{ms}">[MM:SS]</a>`. Clicking any timestamp (summary or transcript) enters Split mode, then scrolls the opposite pane to the matching anchor. If exact match missing, uses closest-by-time fallback. Target highlights `bg-yellow-100` for 1.5s.

**Bidirectional linking:** Summary timestamp → scroll Transcript pane. Transcript timestamp → scroll Summary pane. Both directions resolve to `#t-{ms}` anchors.

**Split mode visual design:** Flow layout with sticky headers. Summary pane uses `bg-white` with sticky "Key Takeaways" header. Transcript pane uses `bg-gray-50` with sticky "Transcription" header. No divider line — separation via background shade difference.

#### Run History Page (`/polls`)

**List view:** Paginated table/list (**25 items/page**, same styling language as Signal Viewer), with columns:
| Column | Content |
|---|---|
| Date / Time | When the poll ran (triggered time) |
| Status | Overall status of the run (`running`, `done`, `failed`) — colored badge |
| New Signals | Count of new signals processed this run |
| Channels | Brief summary: "3/5 done, 1 failed" or similar aggregate |

- **Sort order:** newest first (most recent poll on top).
- Clicking a row navigates to `/polls/:id` showing per-channel breakdown.

#### Admin Panel (`/admin`) — Tabbed Layout
Three tabs: [Channels] [Topics] [Polling]. Alpine.js-managed tab state.

**[Channels] tab:**
- **Add Channel form:** channel_id input + topic dropdown selector (required). On add, fetch and store display name + avatar from YouTube. Topic must exist and be non-NULL for channel to be pollable.
- **WatchList table:** per-channel row with avatar, name, topic badge (shows topic short_name), last-poll date, active/inactive toggle, change-topic dropdown, remove button. Channels with NULL topic shown with warning indicator.

**[Topics] tab:**
- **Add Topic form:** key (slug), short_name (display label), filter_text (textarea — relevance criteria for LLM prompt).
- **Topics table:** per-topic row showing key, short_name, filter_text preview (truncated), channel count, edit button, delete button. Delete uses force-delete: sets channel `topic_id` to NULL rather than cascading delete.

**[Polling] tab:**
- **Poll trigger:** "Run Poll Now" button + lookback days input. Trigger enqueues a `PollRun` job → UI shows progress inline via HTMX polling. Run History list lives on `/polls`.
- **Live Progress Widget:** HTMX periodic `hx-get` with `hx-trigger="every 3s"` and `hx-swap="outerHTML"` for self-replacing progress updates.

**Authentication:** No auth — internal tool, trusted access only. Admin routes guarded by obscurity (URL hidden in nav).

#### Data Model — Sentiment Score
- Overall sentiment is stored numerically on a **1–5 integer scale** for UI display: `1 = Very Negative`, `2 = Negative`, `3 = Neutral`, `4 = Positive`, `5 = Very Positive`. The LLM also returns the original categorical label for reference.

## Anti-Glossary
Do NOT use these terms (use the glossary equivalent instead):
- ~~Video~~ → use **Signal** (the video is the carrier; the Signal is what we process)
- ~~Transcript~~ → use **Transcription**
- ~~Feed~~ → use **Channel** or **RSS feed** (be specific)
- ~~Analysis~~ → use **Summary** or **Sentiment** (be specific)
