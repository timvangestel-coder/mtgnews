# MTG News — Context

## Description
MTG news & sentiment scanner. Ingests YouTube videos from watched channels, extracts transcriptions, and produces AI-powered summaries and sentiment analysis using a local LLM.

## Glossary

| Term | Definition |
|---|---|
| **Signal** | The primary domain object. A unit of raw text (transcription) extracted from a source video, along with its metadata (video ID, channel, title, published date). The Signal is what the system processes — not the video itself. |
| **Processing State** | A `processing_state` field on Signals: `"pending"` (has transcription, LLM has not run), `"irrelevant"` (manually marked as not worth reviewing; hidden from signal list by default), `"summarized"` (LLM ran, summary/sentiment/entities stored). Replaces the previous two-column model (`processed_at` + `relevance_status`). Abort/stale cleanup deletes only `"pending"` signals — `"irrelevant"` and `"summarized"` always survive. All transitions between states are allowed via a dropdown on Signal Detail page (ADR-0013). |
| **Reviewed Flag** | A `reviewed INTEGER DEFAULT 0` column on Signals tracking whether the user has personally opened/read a signal. Independent of `processing_state` — changing processing state does not auto-update reviewed. Used for "Show Unreviewed" filter on Signal List page (default view shows only unreviewed signals). Visual indicator (badge/icon) next to unreviewed signal titles. Separate column preserves semantic clarity: processing_state tracks *system processing*, reviewed tracks *user attention* (ADR-0013). |
| **Channel** | A YouTube channel on the WatchList. Identified by its YouTube channel ID. The system polls Channels for new videos. Each Channel belongs to exactly one Topic (1:N). Channels with no topic assigned (`topic_id IS NULL`) are blocked from polling. |
| **Topic** | A domain category that groups channels sharing a content focus. Defined by four fields: `key` (machine-readable slug, e.g. `mtg`, `ai`), `short_name` (UI display label, e.g. "MTG", "AI"), `filter_text` (the relevance criteria text injected into the LLM prompt), and `summary_prompt` (nullable TEXT — per-topic custom prompt template for the PromptAssembler). Topics are managed independently of Channels via the Admin panel. Deleting a topic sets channel `topic_id` to NULL (force-delete). |
| **WatchList** | The configured set of Channels the system monitors. Stored persistently in SQLite. |
| **Transcription** | The caption/subtitle text extracted from a YouTube video using `yt-dlp`. May be manually authored or auto-generated by YouTube. |
| **Summary** | An AI-generated concise summary of a Signal's topic-relevant content, produced by the local LLM. |
| **Sentiment** | An AI-generated sentiment assessment of a Signal. Two levels: **Overall** (single label for the entire Signal) and **Per-Entity** (sentiment labels for each entity mentioned). Entity topic membership is derived via join chain (`entity_mentions → signals → channels → topics`), not denormalized. |
| **Entity** | A topic-relevant concept mentioned in a Signal: card name, set name, player name, model, company, etc. Extracted and scored by the LLM during sentiment analysis. Entity type vocabulary is free-form (not constrained per-topic). Topic membership derived via join chain through signals→channels→topics. |
| **SignalContext** | A resolved context object returned by `resolveSignalContext(videoId, db)` containing raw data needed for LLM analysis: `transcriptionJson` (raw JSON blob), `topicId`, `filterText`, and `summaryPrompt` (custom template or NULL). The `summaryPrompt` uses a three-tier resolution chain: (1) topic override (`topics.summary_prompt` if NOT NULL), (2) DB global default (`app_settings.default_summary_prompt` via `getAppSetting`), (3) compiled code fallback (`defaultPromptTemplate()`). The interface carries resolved data — no formatting decisions. Created to eliminate ad-hoc SQL queries in `llm.ts`. |
| **AppSettings** | A key/value store in SQLite (`app_settings` table: `key TEXT PRIMARY KEY`, `value TEXT`) for runtime-configurable global defaults. Accessed via `src/db/app-settings.ts` which exports `getAppSetting(db, key): string | null`, `setAppSetting(db, key, value): void`, and `deleteAppSetting(db, key): void`. Currently used for `default_summary_prompt` — the global default prompt template editable via Admin [Topics] tab. Three-tier resolution chain: topic override → DB global default → compiled code fallback (`defaultPromptTemplate()`). Setting an empty string in the UI calls `deleteAppSetting` to clear the DB override and fall back to compiled default. |
| **Ingestion** | The per-candidate pipeline that transforms an RSS candidate into a persisted Signal: caption extraction → segment grouping → DB insert. Produces an `IngestResult` indicating outcome (ingested, duplicate, or no captions). Called by the Poll worker for each new video candidate. |
| **Poll** | A single execution of the scheduled ingestion cycle: fetch RSS feeds for all active Channels with a non-NULL `topic_id`, identify new videos, run Ingestion for each candidate, then run LLM analysis on new signals. Runs once daily. Channels without a topic are skipped. |
| **E2E Test** | End-to-end browser test using Playwright. Spins up the Express server against an in-memory SQLite DB seeded with fixture data. Tests client-side behavior (Alpine.js toggles, HTMX swaps, transcription expand/collapse) that supertest unit tests cannot verify. |
  | **Run Abort** | A user-initiated cancellation of an active PollRun. Fires an in-memory AbortController to cancel in-flight LLM HTTP requests and stop the worker between tasks. Then deletes signals with `processing_state = 'pending'` by poll_run_id — only unanalyzed signals are removed; irrelevant and summarized signals always survive. Always transitions run status to `done-forced` with correct `new_signal_count`. The worker's pool callbacks guard `incrementDone()` with `signal?.aborted` so in-flight tasks that settle after abort cleanup don't inflate progress counters. Aborted tasks on the AbortError path skip `incrementDone()` entirely. Worker waits for full `pool.drain()` before returning (simple, safe — UI reflects abort immediately via DB update). |
  | **Stale Run** | A PollRun stuck at `status='running'` with no `completed_at`, caused by server shutdown mid-execution. Auto-cleaned on startup: signals with `processing_state = 'pending'` deleted by poll_run_id — aligned with abort logic. Run transitions to `done-forced` if processed > 0, deleted entirely if processed == 0. |
  | **Poll Run UI State** | The progress widget in the Admin [Overview] tab shows per-channel counters during an active PollRun. Single streaming model (no phases): each channel step displays one of five states — `fetching` (blue), `X/Y` processing (blue), `Y/Y` done (green), `none` no signals (grey), `failed` (red). Channels are sorted alphabetically by display name (NULLs first) and remain in that order throughout the run. For aborted runs: channels with 0 summaries show `skipped` (grey), partial work shows `X/N` (amber), fully-completed channels stay `N/N` (green). RSS polling is invisible to the UI (stays "fetching"). Failed signals count toward `done`. No global counter banner. See https://github.com/timvangestel-coder/mtgnews/issues/79. |
 | **Route Module** | An Express Router factory (e.g., `createSignalsRouter`) that receives a service layer dependency and mounts HTTP handlers for one domain area. Each module exports a single `Router` instance. Route modules are thin: parse params → call service → render template or redirect. Mounted in `server.ts` via `app.use()`. |
 | **Service Layer** | Domain-specific orchestration modules that sit between route handlers and pure domain functions. Two naming conventions: `-Service` suffix for read-only query services (`SignalQueryService`, `PollQueryService`), `-Manager` suffix for write/CRUD managers (`ChannelManager`, `TopicManager`). The service wraps DB access and coordinates multi-function operations (e.g., query + format + enrich). Services are the primary unit of testing — unit tests target services directly with in-memory SQLite, while route modules get lightweight HTTP smoke tests. |
  | **HTTP Retry Module** | A generic `fetchWithRetry(url, options, retryConfig)` module that handles timeout via AbortController, external abort signal merging, transient error detection (TypeError, ECONNRESET, ECONNREFUSED), and exponential backoff. Consumer-agnostic: the LLM client is the current adapter; other HTTP clients may adopt it later. |
  | **Concurrency Pool** | A deep module (`concurrency-pool.ts`) that enforces a strict upper bound on parallel task execution using a semaphore counter. Interface: `run(fn)` dispatches a task (queues if at limit), `drain()` waits for all dispatched tasks. Guarantees at most `LLM_CONCURRENCY` (default 3) LLM requests are in flight simultaneously. Single global instance shared by Poll analysis and Chat processing — created in `server.ts`, passed to both PollRunManager and ChatQueue. Replaced the original inline `Promise.race`-based pool that allowed burst concurrency exceeding the configured limit. |
   | **Transcription Merge Module** | A pure-function module (`transcription-merge.ts`) containing `mergeOverlappingSegments()` and `groupSegments()`. Input: raw caption segments. Output: grouped `[{time: number (ms), text: string}]`. No I/O, no format detection — purely algorithmic. Tested in isolation via `merge-segments.test.ts`. |
   | **SignalChat** | A Q&A feature supporting two scopes using an async job pattern matching Poll Runs. Stored in `signal_chat` table (`id`, `signal_video_id TEXT (nullable)`, `question`, `answer TEXT (nullable)`, `created_at`, `topic_key TEXT (nullable)`, `channel_id TEXT (nullable)`, `include_irrelevant INTEGER DEFAULT 0`). Two-phase lifecycle: (1) submit — INSERT with `answer=NULL`, return immediately; (2) process — background worker dispatches via global ConcurrencyPool, UPDATE on success. Failed questions remain with `answer=NULL`. No real-time streaming — **manual JS polling** (`setInterval` every 3s in Alpine component `_startStatusPolling()`) checks pending rows. Replaced HTMX-based `hx-trigger="every 2s"` to eliminate timer accumulation bugs during outerHTML swaps on list-scoped chat with many pending messages. **Two scopes:** (a) Per-signal: `signal_video_id` set, filter columns NULL — panel on Signal Detail page (~380px slide-in). (b) List-scoped: `signal_video_id=NULL`, filter columns set — panel on Signal List page (760px slide-in). History strictly separated by scope type. Q&A ratio displayed in Signal List as `COUNT(answer IS NOT NULL) / COUNT(*)`. **Scope-aware history reload:** chat panel tracks `_lastHistoryScope` and re-loads history when user closes chat, changes filter pills, and re-opens — ensuring questions/answers always reflect current filters. Supports ChatCancel for aborting in-flight LLM work. |
   | **ChatCancel** | User-initiated cancellation of a pending SignalChat question. Fires an AbortController to cancel the in-flight LLM HTTP request via the `abortSignal` seam in `llm.ts`. Implemented as `chatQueue.cancel(id)` — a two-step: (1) fire abort(), (2) immediately DELETE FROM signal_chat. The in-flight task catches AbortError and returns silently; if the response arrived before abort took effect, `signal?.aborted` is checked before UPDATE so answers are never persisted for cancelled questions. Mirrors the Run Abort pattern used by PollRunManager but operates per-question instead of per-run. Controllers stored in a `Map<number, AbortController>` registry on ChatQueue, cleaned up when tasks settle via finally block. |
| **AgentChat** | A two-step retrieval pattern for all chat questions using LLM-driven tool calling instead of embedding-based RAG. **Round 1:** Send signal index (title + summary per signal in scope, ~50 tokens/signal) + question → LLM reads summaries, decides which signals are relevant, calls `get_compact_text(videoIds)` tool. **Tool execution:** Backend fetches compact_text from SQLite for requested videoIds, injects result as `role: "tool"` message. **Round 2:** LLM generates final answer with retrieved compact texts in context. Supports up to 3 rounds (hard cap) if LLM needs additional signals after reading first batch. Retrieval reasoning is streamed to UI but NOT persisted — only the final answer goes into `signal_chat.answer`. **Loop consolidation:** `_runAgentLoop()` is the single private agent loop implementation. Both per-signal and multi-signal scopes delegate through thin adapters that resolve index scope via `resolveIndexScope()` uniformly, then call `_runAgentLoop()`. An empty-answer guard persists a fallback message if all rounds produce only tool calls without a final answer. **Deferred deepening:** The string-based history interface in the agent loop is intentional — deepening the LLM seam to accept structured message arrays is deferred so future readers do not mistake it as an oversight. |
    | **AgentChatTool** | The tool schema exposed to the LLM during AgentChat: `{"name": "get_compact_text", "parameters": {"videoIds": ["string"]}}`. No format parameter (always returns compact_text). No cap on videoIds per call — trusts the LLM to request only relevant signals. Implemented as an in-process SQLite query in ChatManager, not an external service. The tool response includes `{videoId, title, content}` per requested signal. |
    | **ChatManager** | A `-Manager` suffix service (`src/services/chat-manager.ts`) supporting AgentChat via polymorphic scope resolution and tool calling. Core operations: `submit(scope, question)` inserts a pending row; `process(id)` reads scope from DB row → calls `resolveScope(scope)` for lightweight index data (title + summary only) → fetches Q&A history → assembles agent prompt → runs multi-round LLM loop with `get_compact_text` tool handler → UPDATE on success. The tool handler is an in-process SQLite query: receives videoIds from LLM, returns compact_text from DB as `role: "tool"` message. Supports up to 3 retrieval rounds per question. Retrieval reasoning streamed to UI but excluded from persisted answer. Exports: `getHistory(signalVideoId?, topicKey?, channelId?)`, `submit()`, `delete(id)`. **Transform seam:** ChatResponseFormatter applied before persisting for all answers. Always receives a signalMap — single-signal passes one entry, multi-signal passes many. **Rendering seam:** Persisted answers are pre-formatted HTML rendered via `<%- msg.answer %>` (unescaped). Pending rows show "processing..." spinner via manual JS polling. |
   | **Generated Title** | An AI-generated concise title for a Signal, produced during Signal analysis via the same LLM call that generates Summary + Sentiment. Stored in `signals.generated_title` column (TEXT, nullable, max 100 chars). Displayed in Signal List instead of the truncated Summary column. Falls back to original `signals.title` when NULL. Prompt instructs "max 100 characters"; application truncates as safety net. |
      | **CompactTranscription** | A telegraphically compressed version of a Transcription, produced by the LLM during Signal analysis via the same call that generates Summary + Sentiment. Created by removing filler words, function words (articles, prepositions, auxiliary verbs), and punctuation while keeping content words (nouns, main verbs, adjectives, proper names, numbers) and preserving `[T:ss]` timestamp markers. Instruction-based compression — the LLM judges what to remove rather than using a hardcoded stopword list. Negation words ("not", "no", "never") are preserved since they change meaning. Stored in `signals.compact_text` column (TEXT, nullable). Used by AgentChat as the retrieval target via the `get_compact_text` tool — the LLM requests compact_text for relevant signals rather than receiving all compact_text upfront. Replaces the previous direct-injection model where all scope compact_text was sent in a single prompt. |
| **SignalIndexEntry** | A lightweight representation of a signal for the AgentChat index: `{ videoId: string, title: string, summary: string }`. ~50 tokens per entry when serialized to XML. Produced by `resolveIndexScope()` for AgentChat — contains only metadata needed for the LLM to decide which signals to retrieve via tool calls. Excludes compact_text and transcription. Replaces the heavy `ChatSignalContext` in AgentChat scope resolution. **Ordering convention:** Both `resolveScope()` and `resolveIndexScope()` use `published_at DESC` ordering consistently, ensuring the signal index presents newest signals first regardless of scope type. |
     | **ChatContext** | A resolved context object for chat prompts: `{ transcriptionJson, summary, history: [{question, answer}], question }`. Passed to `assembleChat()` in PromptAssembler. History length controlled by ChatManager (caller), not the assembler — assembler is a pure formatter with no token-budget awareness. |
   | **LLM Streaming Seam** | The split in `llm.ts` creating two adapters at the HTTP seam: `callLlmSync(config, prompt) → Promise<string>` for sync JSON responses (analysis), and `callLlmStream*(config, prompt): AsyncGenerator<string>` for streaming token chunks (chat). Both share internal HTTP logic but diverge at response consumption (`.json()` vs `ReadableStream`). The seam enables testing via mock adapters without mocking `fetch`. Existing `analyzeSignal()` remains unchanged (strangler pattern — new clean functions coexist with legacy coupled function). |
    | **ChatResponseFormatter** | A deep module (`chat-response-formatter.ts`) that transforms raw LLM text into rendered HTML for chat answers. Interface: `format(text, signalMap) → string`. Handles all timestamp link formats uniformly: converts `<<videoId:T:ss>>` citation delimiters and bare `[T:ss]`/`[MM:SS]` timestamps to `<a href="/signals/{videoId}#t-{ms}" data-timestamp="{ms}" data-video-id="{videoId}" class="pill">{title} · [MM:SS]</a>`. Also converts Markdown (`**bold**`, `*italic*`) to HTML. Takes a required `signalMap: Record<videoId, {title}>` — always provided by ChatManager (single-signal passes one entry, multi-signal passes many). No DB access — pure string transformation. Replaces the previous two-module design (CitationFormatter + TimestampFormatter) that produced incompatible link formats causing cross-page timestamp navigation bugs. |
      | **ChatScope** | A polymorphic scope object identifying the context for chat questions: `{ videoId?: string, topicKey?: string, channelId?: string, includeIrrelevant?: boolean }`. When `videoId` is set → per-signal chat. When filter fields are set → list-scoped multi-signal chat. Persisted in `signal_chat` row columns so scope survives the async queue pipeline (enqueue → DB → process). |
      | **ResponseFormat** | A data map (`FORMAT_INSTRUCTIONS`) inline in `prompt-assembler.ts` providing LLM response format instructions as strings keyed by style (`'annotated-index' | 'plain'`). Style selected via AppSettings key `chat_response_format`. Output injected into chat prompt templates via `{FORMAT_INSTRUCTIONS}` placeholder. The `'annotated-index'` style instructs the LLM to produce: (1) a 1-sentence summary, (2) per-source `### Source Title` headings followed by markdown tables with `Timestamp | Finding` columns using `T:ss` format, and (3) thematic tags on one line. Same instructions used for both single-signal and multi-signal chat — the LLM produces one section vs multiple sections. Standalone module deferred until format instructions grow beyond trivial string lookup. Citation pills inside table cells are protected via `\x00` placeholder dance before `marked` processes text, then restored afterward. |
      | **MarkdownRenderer** | The markdown-to-HTML conversion seam in ChatResponseFormatter, implemented by the `marked` library with GFM enabled (`{ gfm: true, breaks: false }`). Runs after citation pill generation — pills are protected via `\x00` placeholder dance before `marked` processes text, then restored afterward. Replaces the original 4-regex `markdownToHtml()` function that only handled bold/italic. |
      | **LLMPhase** | A three-phase model describing real-time LLM generation activity: `'intake'` (request sent, no SSE chunks yet — fired at request-send time), `'reasoning'` (model generating Chain-of-Thought via `delta.reasoning_content`), `'answering'` (model generating visible answer via `delta.content`), `'done'` (`finish_reason: "stop"`). Distinct from Signal processing state and Poll run step status. Detected by SSE field presence in the streaming variant of the LLM call. |
| **PhaseRegistry** | A generic in-memory module (`src/phase-registry.ts`) providing `Map<K, { phase: LlmPhase, tokenCount: number }>` for tracking live LLM generation phases. Instantiated per consumer via generics: `PhaseRegistry<number>` on ChatQueue (keyed by question id), `PhaseRegistry<string>` on PollRunManager (keyed by videoId). Written by the `onPhaseChange` callback in the streaming LLM call. Read by status polling endpoints. Entries cleaned up when tasks settle. Level 2 module pattern matching ConcurrencyPool — one deep module, separate instances per consumer. Extended with optional `round?: number` field for AgentChat multi-round transparency. |
| **LLM Streaming Seam** | The split in `llm.ts` creating two adapters at the HTTP seam: `callLlmSync(config, prompt) → Promise<string>` for sync JSON responses (analysis), and `callLlmStream*(config, prompt): AsyncGenerator<string>` for streaming token chunks (chat). Extended with `callLlmStreamWithPhases*` which adds an optional `onPhaseChange(phase: LlmPhase, tokenCount: number, round?: number)` callback firing at phase transitions. The generator yields only content tokens (reasoning_content is internal — used for phase detection and counting but not yielded). Both share internal HTTP logic but diverge at response consumption (`.json()` vs `ReadableStream`). The seam enables testing via mock adapters without mocking `fetch`. |
    | **ScopeSource** | A deep module (`scope-source.ts` / `views/scripts/scope-source.js`) providing pure functions for reading and building ChatScope data from URL query params. Interface: `fromCurrentURL() → {topicKey?, channelId?, includeIrrelevant}` reads browser URL; `buildHistoryURL(scope) → string` constructs `/chat/history?...`; `buildAskBody(question, scope) → AskBody` builds POST body for `/chat/ask`. Framework-agnostic (no Alpine, no HTMX). Eliminates Alpine state drift by providing point-of-use scope reading instead of maintaining stale reactive properties. Loaded as `window.ScopeSource` before `chat-panel.js`. Tested via unit tests (`scope-source.test.ts`, 17 tests), E2E tests (`tests/e2e/chat-scope.spec.ts`, 10 tests), and integration tests (`src/routes/chat-router-scope-roundtrip.test.ts`, 10 tests). |
     | **DateFilter** | A preset date range filter (`'all'` \| `'last-week'` \| `'last-month'` \| `'last-year'`) applied independently to both the signals table and chat scope resolution. URL param: `date=last-week`. Stored in `signal_chat.date_filter` column (TEXT, default `'all'`) for chat history scoping. The `computeDateRange(dateFilter)` pure function maps presets to ISO date bounds (`{ from?: string }`) at request time using JS `new Date()` arithmetic — frozen-at-request-time is acceptable. When `date='all'` or absent in the URL, no date condition is applied: all signals and all chat conversations are shown regardless of their stored `date_filter` value (same "show-all" pattern as topic/channel filters). Date filtering reaches scope resolution via a separate options parameter on `resolveScope()` / `resolveIndexScope()`, keeping `ChatScope` identity-pure. |
 | **TimestampNav** | A framework-agnostic JS module (`views/scripts/timestamp-nav.js`) providing click interception for timestamp pill links across all pages. Interface: `init({ samePage, onSummaryClick?, onTranscriptClick?, onChatClick? })`. When `samePage=true`, clicks are intercepted and the appropriate callback fires with `{ms}`. When `samePage=false`, clicks pass through to browser navigation (href follows). Installs one document-level click delegate using CSS selectors: `#summary-pane a[data-timestamp]`, `#transcript-pane a[data-timestamp]`, `[data-chat-panel]/.chat-history a[data-timestamp]`. Loaded as `window.TimestampNav` before Alpine. Follows the ScopeSource pattern — pure JS module, no framework dependencies. |
     | **RssFeedFetcher** | A deep module that replaces the raw `fetchUrl()` / `fetchRssSync()` functions in `rss-discovery.ts`. Interface: `async fetch(channelId) → RssFetchResult { xml, status }` throwing `RssError`. Provides HTTP status awareness (rejects non-2xx), typed errors, per-channel backoff with global IP-level cooldown, and inter-request delay. Hybrid backoff: per-channel for persistent failures + global row (`__global__`) triggered by any 429 blocking all channels. Backoff resets immediately on first success. HTTP 404 uses hybrid logic: established channels (`ever_successful=true`) treated as transient (backoff), new channels fail hard after 2 attempts. Inter-request delay defaults to 1s via `POLL_REQUEST_DELAY_MS` env var. Exponential backoff: 30s → 1m → 2m → 5m cap. State stored in `rss_backoff` table. See ADR-0017. |
     | **DesignToken** | A semantic color/value mapping defined in `views/scripts/tailwind-config.js` that replaces ad-hoc Tailwind color classes across all view files. Six tokens: `brand` (#6366f1, primary actions), `success` (#10b981, positive/completed), `warning` (#f59e0b, new/aborted), `danger` (#ef4444, failed/irrelevant), `muted` (#64748b, inactive/borders), `surface` (#f8fafc, cards/panels). Config loaded via `<script type="module">` before Tailwind CDN in `layout.ejs`. Zero build step — no Tailwind CLI migration. Creates utility classes like `bg-brand-500`, `text-success-600`. See ADR-0018. |
     | **UiStateModule** | A centralized Alpine.js data module (`views/scripts/ui-state.js`) exporting named functions that return Alpine.js data objects: `signalListView()` (card list state), `filterBar()` (topic/date/channel filter visual state from URL params), `chatPanel()` (open/close, streaming target), `detailTabs()` (summary/transcript/split view). Replaces inline `x-data="{...}"` blocks scattered across EJS templates. Templates reference functions: `x-data="filterBar()"`. Loaded as `window.UiState` before Alpine.js. See ADR-0018. |
     | **RssError** | A typed error class thrown by RssFeedFetcher with `statusCode` and `channelId` properties. Static guards: `isRateLimited(e)` for 429, `isBadChannel(e)` for 404 on never-successful channels, `isTransient(e)` for 5xx/network errors or 404 on established channels. Replaces the previous pattern where parse failures swallowed the real HTTP status code. |
| **DarkModeModule** | A small JS module (`views/scripts/dark-mode.js`) that auto-detects `prefers-color-scheme` on page load, toggles `<html class="dark">`, and persists user preference in `localStorage`. Sidebar provides a sun/moon toggle button for manual override. Works with `darkMode: 'class'` in tailwind-config.js. All view elements use `dark:` variants on background/text classes. See ADR-0019 Decision 2. |

     | **MotionToken** | A unified transition duration of **200ms** applied to all UI transitions, replacing the previous ad-hoc durations (`duration-1000`, `duration-300`, inline `0.15s`).      | **KeyboardNav** | Full keyboard accessibility pattern applied across all interactive UI elements: roving tabindex on filter pills (ArrowLeft/Right), `role="tablist"` with arrow keys on detail tabs, Escape-to-close + focus trap in chat panel, proper `aria-*` attributes throughout.      | **ChatVisualPolish** | The visual upgrade for the chat panel: message bubbles (user left-aligned with grey avatar + white container, AI right-aligned with brand avatar + light brand container), gradient header (brand to purple), typing indicator with bouncing dots replacing raw "processing..." text, phase micro-labels below dots.      | **ZeroInlineStyles** | A hard rule: all styling flows through token-based Tailwind classes or CSS custom properties defined in config modules. Colors use token classes (`text-warning-500` instead of `color: #eab308`). See ADR-0019 Decision 8. |
    | **ChatSignalContext** | A resolved context object for multi-signal chat: `{ signalContext: SignalContext, videoId: string, title: string, channelDisplayName: string }`. Wraps SignalContext with display fields needed for citation pills and scope badges. Produced by `resolveScope(scope)` in ChatManager. Keeps SignalContext domain-pure (no display concerns). |
| **PromptAssembler** | A deep module (`prompt-assembler.ts`) that renders LLM prompts from XML templates. Three entry points: `assemble(context: SignalContext) → string` for signal analysis, `assembleChat(context: ChatContext, customTemplate?) → string` for per-signal chat, and `assembleMultiSignalChat(context) → string` for list-scoped multi-signal chat. All use three-tier template resolution (topic/DB override → AppSettings global default → compiled fallback). Templates use `{PLACEHOLDER}` syntax inside XML tags (`<transcription>`, `<summary>`, `<conversation_history>`, `<signals>`). Pure functions — no DB access, no side effects. `formatTranscription(json)` shared utility for all paths. Multi-signal template uses `<signal video_id="...">` XML blocks per signal and instructs LLM to use `<<videoId:T:ss>>` citation format. |
| **PhaseVisibility** | The architecture that makes LLM phase transitions observable in the chat UI during processing. Composed of: (1) `PHASE_BATCH_SIZE=1` in ChatQueue — every `onPhaseChange` callback schedules a `setTimeout(() => {}, 0)` macrotask yield so the browser gets repaint opportunities between synchronous phase bursts; (2) `_chatAnswerStatus.ejs` renders phase labels with `data-chat-phase`, `data-chat-token-count`, and `data-chat-round` HTML attributes on every poll cycle; (3) CSS opacity transition (`transition: opacity 0.15s ease`) on `.chat-phase-text` class in `layout.ejs`, with `.phase-fading { opacity: 0 }` class triggered by JS polling when phase changes — smooth fade-out, text swap, fade-in pattern; (4) round indicators via `PhaseEntry.round` field — UI shows "(Round 2)" when `round > 1`, incrementing each time intake fires after existing entries (new agent loop iteration); (5) `views/scripts/chat-panel.js` `_startStatusPolling()` reads phase/tokenCount/round from JSON status endpoint and updates DOM with fade animation on phase label changes. PollRunManager is unaffected — it polls phases from DB after completion, not during processing. |

## Decisions

### UI Polling Seam — Manual JS vs HTMX

HTMX-based status polling (`hx-trigger="every 3s"` on pending answer divs) was replaced with manual JavaScript `setInterval` polling due to timer accumulation bugs in HTMX 2.x during outerHTML swaps. When list-scoped chat had many pending messages, each element created its own HTMX timer that accumulated during DOM replacement, causing rapid-fire status requests (every ~100ms instead of every 3s). The JS approach uses a single `setInterval` that queries all `[data-chat-status="pending"]` elements and stops cleanly on completion or chat close.

### Acquisition
- **YouTube only** for now, architecture designed for additional channels later.
- **yt-dlp** extracts caption/subtitle tracks from YouTube. No API key required. If a video has no captions available, it is skipped and logged.
- **YouTube RSS feeds** (`/feeds/videos.xml?channel_id=...`) for discovering new videos from watched Channels. No API key required. Returns latest ~15 videos per channel.
- **Shorts exclusion:** YouTube Shorts are filtered out at RSS discovery by detecting `/shorts/` in the entry's link href. Shorts never become candidates and are silently skipped — no counter, no tracking.

### Processing
- **Local LLM** via LM Studio at `http://127.0.0.1:1234/v1/chat/completions` (OpenAI-compatible endpoint).
- **Model:** `qwen/qwen3.6-27b`.
- Each Signal produces: Summary + Overall Sentiment + Per-Entity Sentiment, generated via a **single merged LLM call** returning one JSON response containing all outputs plus a `"relevant"` boolean. Transcription is sent once per signal, not thrice.
- **Relevance filtering:** The LLM receives the channel's Topic `filter_text` as part of the prompt (resolved via `channels.topic_id → topics`) and must first judge whether the content matches. If irrelevant, it returns `"relevant": false`. The system sets `processing_state = 'irrelevant'`, skips storing summary/sentiment/entities. Relevant signals get full analysis stored with `processing_state = 'summarized'`.
- **Generic analyst role:** The LLM prompt uses a generic role ("You are a content analyst") rather than a domain-specific one. Domain scoping is handled entirely through the Topic's `filter_text`. Entity types are free-form (not constrained per-topic).
- **Prompt Template System:** Per-Topic custom prompts stored in `topics.summary_prompt` (TEXT, nullable). Templates use XML placeholder tags (`<transcription>`, `<filter_text>`) that the PromptAssembler injects at render time. When NULL, a compiled default template is used. Templates edited via Admin [Topics] textarea — no code deploy required for prompt iteration. The assembler formats raw transcription JSON into timestamped text before injection. Module boundary: SignalContext resolves data (raw), PromptAssembler renders (pure function of context → string).

### Persistence
- **SQLite** for all data: WatchList (channel IDs), processed Signals (video IDs, transcriptions, summaries, sentiment), and Entity mentions.
- Deduplication by YouTube video ID — each video is processed at most once.
- **Schema initialization** uses simple `CREATE TABLE IF NOT EXISTS` in a single module called at app startup — no migration runner framework. Greenfield project with stable schema.

### Polling & Background Worker Model
- **Queue-based async polling via SQLite job table.** Trigger enqueues a `PollRun` row → background worker (separate file/module, runs in-process but on its own loop) picks up the job → processes channels sequentially (RSS fetch + caption extraction), then runs LLM analysis through a **Concurrency Pool** controlled by `LLM_CONCURRENCY` env var (default 3). The pool uses a semaphore counter to guarantee at most N tasks run simultaneously. Progress tracked in `poll_run_progress` table. UI polls for status via HTMX periodic `hx-get`.
- **Scheduled polling** also creates `PollRun` entries — triggers once daily via `node-cron` or equivalent, runs the same worker path as manual trigger (single source of truth).

### Data Model — Transcription Storage
- Captions from yt-dlp include per-segment timing. Store transcription as a **JSON blob in a SQLite TEXT column** (`TEXT NOT NULL`) containing an array of `{time: number, text: string}` objects. The `time` field is the group start timestamp in milliseconds, rounded to the nearest second. One JSON blob per Signal.
- **YouTube caption deduplication:** YouTube auto-captions produce overlapping "paint-on" segments (contiguous pairs where text is a superset) plus word-level overlap at segment boundaries. `mergeOverlappingSegments()` performs two-phase cleanup: (1) merge contiguous superset segments, (2) trim overlapping leading words at segment boundaries.
- **Transcription grouping:** After merging, raw segments are grouped into ~10-second windows. Segments within the same 10-second window are concatenated into a single group with one `time` value (the rounded start of the first segment in the group). Grouping happens at ingestion time; the grouped result is stored in the DB, raw segments are discarded.
- **Timestamp display:** Transcription is rendered with `[MM:SS]` format, one timestamp per group. Anchor IDs use the group's `time` value in milliseconds (e.g., `id="t-45000"`).
- **LLM summary timestamps:** The LLM receives the grouped transcription with `[T:ss]` timestamp references (one per ~10-second group). Since the LLM sees the same grouped data stored in the DB, its `[T:ss]` references map directly to stored group timestamps. `injectTimestampAnchors()` converts these to `[MM:SS]` display labels and millisecond anchor IDs (e.g., `[T:45]` → `[00:45]` linking to `#t-45000`).

### Schedule
- **Once daily** polling interval (triggered by cron-like scheduler, same code path as manual poll trigger).

### Tech Stack
- **Node.js / TypeScript** (backend + server-side code)
- Express (TypeScript, with `ts-node-dev` or `tsx` for dev hot-reload)
- EJS template engine (`@types/ejs`) — minimal logic, `include()` for layout partials
- HTMX for AJAX-driven HTML swaps (pagination, channel filter, poll trigger, detail navigation)

Use `x-init="htmx.process($el)"` on the container `<div>` inside each template so HTMX re-scans and registers `hx-*` attributes after Alpine inserts them.
- Tailwind CSS via CDN (`<script src="https://cdn.tailwindcss.com">`) for dev; production build optional later
- SQLite via `better-sqlite3` (synchronous, native bindings)
- `yt-dlp` via Node.js child_process or wrapper
- RSS parsing via standard XML parser
- HTTP client for LM Studio OpenAI-compatible API
- Vitest + supertest for backend unit/integration tests
- Playwright for E2E browser tests (local only, in-memory SQLite fixture per suite)

### E2E Testing Strategy
- **Playwright** for browser-level tests: verifies Alpine.js toggle pills, HTMX fragment swaps, Signal Detail three-state layout transitions, bidirectional timestamp scrolling
- **In-memory SQLite fixture** spun up in test lifecycle hooks -> each suite gets fresh seeded DB + server on random port -> no shared state, no flaky tests
- **Scope** (priority order): 1) Signal Viewer, 2) Signal Detail, 3) Admin Panel, 4) Run History
- **Local only** for now; CI integration possible later
- **DOM-Only Assertions:** E2E tests must verify behavior through observable DOM state (visibility, CSS classes, text content, scroll position) — never through framework component state (`Alpine.$data()`, HTMX internals, etc). Use page object modules (e.g., `tests/e2e/fixtures/signal-detail-page.ts`) that expose behavior-level methods (`setViewState`, `clickSummaryPill`, `getPaneHeights`) rather than raw locator access.

### Navigation Structure (Four Top-Level Pages)

| Page | Route Pattern | Purpose |
|---|---|---|
| **Signal Viewer** | `/signals` | Paginated signal list + hierarchical topic→channel filter bar. Clicking a row navigates to that Signal's detail page. |
| **Signal Detail** | `/signals/:id` | Full view of one Signal with a three-state layout (SummaryOnly → TranscriptOnly → Split 30/70). Timestamp links rendered as pill badges. Bidirectional navigation between Summary and Transcription panes. See detailed spec below. |

Navigation between Signal Viewer, Signal Detail, Run History, and Admin Panel is full page transitions (not HTMX fragment swaps) — only the signal table rows within `/signals` are swapped via HTMX for pagination/filtering to keep the toggle pills static.

#### Topic + Channel Filter UI — Hierarchical Toggle Pills
- **Two-row filter bar** above Signal table: Row 1 = Topic pills (All + per-topic), Row 2 = Channel pills (All + per-channel).
- Topic pills use same toggle pill styling as channel pills. Default state: no topic selected -> show all signals, all channel pills visible.
- When a topic is selected: channel pills filter to only channels in that topic. Signal table filters by both topic and channel.
- Powered by Alpine.js store tracking `selectedTopic` + `selectedChannel`. On change, HTMX fires debounced `hx-get` to reload signal table fragment only.

#### Signal List View
- Paginated table/list (**25 items/page**) with columns: published date, time, **Q&A ratio** (format "X/Y" — answered/total from `signal_chat`), **Title** (Generated Title or original title as fallback), sentiment score (numeric 1–5 scale with **red→green** gradient background coloring based on score value — `bg-red-600` for 1 through `bg-green-600` for 5).
- **Sort order:** newest first by processing time (`created_at DESC`).
- **Hierarchical filter bar** above the table: topic pills + channel pills as described above, powered by Alpine.js store. HTMX swap on selection change reloads only the signal table fragment.

#### Signal Detail View (full page at `/signals/:id`)

**Layout:** Single column, centered within the main content area (max-width ~800px). Three-state view system managed by Alpine.js (`viewState: 'summary' | 'transcript' | 'split'`).

1. **Header bar:** Signal title (or "No title"), channel pill badge, published date/time.
2. **Key Takeaways section:** Bullet list of AI-generated key takeaways. Inline timestamp links rendered as **pill badges** (`bg-indigo-100 text-indigo-700 rounded`). Clicking enters Split mode.
3. **Full Transcription section:** Renders transcription segments grouped by ~10-second windows. Segment timestamps are also clickable pill badges enabling bidirectional navigation.

**Three-state layout:**
| State | Summary height | Transcript height | Trigger |
|---|---|---|---|
| `summary` (default) | 100% visible | hidden | Page load, or "Back to Summary" button |
| `transcript` | hidden | 100% visible | "Show Full Transcription" button |
| `split` | ~30% of container | ~70% of container | Any timestamp link click |

All transitions animate over 1s via CSS. Both sections always rendered in DOM (no `x-show` collapse).

**Timestamp linking mechanism:** `injectTimestampAnchors()` converts LLM's `[T:ss]` to pill-styled `<a href="#t-{ms}">[MM:SS]</a>`. Clicking any timestamp (summary or transcript) enters Split mode, then scrolls the opposite pane to the matching anchor. If exact match missing, uses closest-by-time fallback. Target highlights `bg-yellow-100` for 1.5s.

**Bidirectional linking:** Summary timestamp → scroll Transcript pane. Transcript timestamp → scroll Summary pane. Both directions resolve to `#t-{ms}` anchors.

**Split mode visual design:** Flow layout with sticky headers. Summary pane uses `bg-white` with sticky "Key Takeaways" header. Transcript pane uses `bg-gray-50` with sticky "Transcription" header. No divider line — separation via background shade difference.

#### Run History Page (`/polls`)

**List view:** Paginated table/list (**25 items/page**, same styling language as Signal Viewer), with columns:
| Column | Content |
|---|---|
| Date / Time | When the poll ran (triggered time) |
| Status | Overall status of the run (`running`, `done`, `failed`) — colored badge |
| New Signals | Count of new signals processed this run |
| Channels | Brief summary: "3/5 done, 1 failed" or similar aggregate |

- **Sort order:** newest first (most recent poll on top).
- Clicking a row navigates to `/polls/:id` showing per-channel breakdown.

#### Admin Panel (`/admin`) — Tabbed Layout
Four tabs: [Overview] [Channels] [Topics] [Settings]. Alpine.js-managed tab state.

**[Channels] tab:**
- **Add Channel form:** channel_id input + topic dropdown selector (required). On add, fetch and store display name + avatar from YouTube. Topic must exist and be non-NULL for channel to be pollable.
- **WatchList table:** per-channel row with avatar, name, topic badge (shows topic short_name), last-poll date, active/inactive toggle, change-topic dropdown, remove button. Channels with NULL topic shown with warning indicator.

**[Topics] tab:**
- **Add Topic form:** key (slug), short_name (display label), filter_text (textarea — relevance criteria for LLM prompt).
- **Topics table:** per-topic row showing key, short_name, filter_text preview (truncated), channel count, edit button, delete button. Delete uses force-delete: sets channel `topic_id` to NULL rather than cascading delete.

**[Overview] tab:**
- **Poll trigger:** "Run Poll Now" button + lookback days input. Trigger enqueues a `PollRun` job → UI shows progress inline via HTMX polling. Run History list lives on `/polls`.
- **Live Progress Widget:** HTMX periodic `hx-get` with `hx-trigger="every 3s"` and `hx-swap="outerHTML"` for self-replacing progress updates. Shown only when a run is active.
- **Status counts:** active Channels, Topics, summarized Signals, Pending signals.
- **Recent Runs:** last 5 poll runs with status icons and links to `/polls/:id`.

**Authentication:** No auth — internal tool, trusted access only. Admin routes guarded by obscurity (URL hidden in nav).

#### Data Model — Sentiment Score
- Overall sentiment is stored numerically on a **1–5 integer scale** for UI display: `1 = Very Negative`, `2 = Negative`, `3 = Neutral`, `4 = Positive`, `5 = Very Positive`. The LLM also returns the original categorical label for reference.
