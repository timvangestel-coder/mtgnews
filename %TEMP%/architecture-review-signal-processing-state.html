Problem:

"Processed" is encoded across TWO columns with overlapping meaning.

  • processed_at IS NULL means "not summarized" AND "not yet analyzed"
  • relevance_status = 'irrelevant' means "analyzed but irrelevant"
  • A signal can be processed (LLM ran) yet still match processed_at IS NULL
Callers must know BOTH columns:
poll-run-manager.ts: DELETE WHERE processed_at IS NULL
query.ts: relevance_status != 'irrelevant'
stale-run.ts: processed_at IS NULL
llm.ts: sets one or the other

Solution:

One column. Three values. Every caller reads one thing.

  • processing_state = 'pending' → not yet analyzed
  • processing_state = 'irrelevant' → LLM ran, said no
  • processing_state = 'summarized' → LLM ran, summary stored
All callers read ONE column:
DELETE WHERE processing_state = 'pending'
SELECT WHERE processing_state != 'irrelevant'
UPDATE SET processing_state = 'summarized'

Problem

The "is this signal processed?" question is split across two columns with contradictory semantics — processed_at lies for irrelevant signals, forcing every caller to also check relevance_status.

Solution

Replace processed_at + relevance_status with a single processing_state TEXT column (pending | irrelevant | summarized). The timestamp moves to a dedicated analyzed_at INTEGER if callers need it (currently only used for sort order, which created_at already covers).

Wins

  • Locality: "is processed?" logic lives in one column, not scattered across N callers
  • Leverage: abort cleanup becomes WHERE processing_state = 'pending' — one predicate, zero ambiguity
  • Test surface shrinks: tests verify one enum, not two-column state machines
  • Stale run alignment: same predicate as abort — no duplication drift
  • CONTEXT.md consistency: glossary says "marks processed_at" for irrelevant; column name matches intent

Abort cleanup — Before vs After

BEFORE — fragile double-negative
DELETE FROM signals
WHERE poll_run_id = ?
  AND processed_at IS NULL
-- BUG: also deletes irrelevant!
AFTER — explicit positive
DELETE FROM signals
WHERE poll_run_id = ?
  AND processing_state = 'pending'
-- Only unanalyzed signals.

One module owns transitions. Callers use isPending(), markSummarized(), canDeleteOnAbort().

Problem

Signal state transitions are inline SQL scattered across 4 modules — no locality for the "what can transition where" rules.

Solution

A pure module that exposes: isPending(state), markSummarized(db, videoId), markIrrelevant(db, videoId), pendingForRun(db, runId). All SQL for signal state lives here.

Wins

  • Locality: transition rules in one module, not 4 files
  • Leverage: tests cover transitions through one interface
  • Deletion test passes: deleting the module would scatter SQL across callers again

Top recommendation

Unify Signal Processing State into a single column

This is the root cause of the abort cleanup bug. Every other symptom (stale run alignment, CONTEXT.md contradiction, query.ts double-negative) flows from this one architectural decision: encoding "processed" across two columns with overlapping semantics.

The fix is a column rename + migration that costs one SQL ALTER but pays back in every caller that currently needs to reason about both processed_at and relevance_status. The Signal State module (Candidate 2) is the natural follow-up once the column shape is correct.
