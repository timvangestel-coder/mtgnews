Problem

All 5 E2E tests in signal-chat-overlay.spec.ts timeout waiting for the chat toggle button [data-chat-toggle] to become visible. The button uses Alpine's x-show="!chatOpen" where chatOpen: false, so the button should be visible (!false = true). Yet Playwright confirms display: none persists after 15 seconds.

629 unit tests pass. 40 of 45 E2E tests pass. Only these 5 fail.

Candidate 2: Unnest ChatPanel from Signal Detail x-data

Strong

Files

views/signal-detail.ejs, views/_chatPanel.ejs

Problem — Tightly Coupled Nesting

The chat panel's x-data is physically nested inside the signal-detail's x-data. This creates a coupling seam where the inner component's lifecycle depends on the outer one. The chatPanel() function is a pure adapter (it takes scope, returns state) but the nesting forces it to inherit the outer component's initialization timing.

Deletion test: If you delete the nesting and place the chat panel outside the outer x-data, the initialization order problem disappears entirely. The complexity was in the nesting, not in either component individually.

Solution

  1. Move the <%- include('_chatPanel'...) %> to be OUTSIDE the closing
of the signal-detail's outer x-data.
  • The chat panel becomes a sibling Alpine component rather than a child, giving it independent initialization.
  • This also applies to signals.ejs where the multi-signal chat panel is already outside any x-data (so it works there — confirming the nesting theory).

    • Benefits

    • Locality: The chat panel becomes a self-contained module with its own Alpine lifecycle. No dependency on the parent component's init order.
    • Leverage: Fixes the root cause (nested initialization) rather than working around symptoms.
    • Tests improve: Independent x-data means Playwright can wait for the chat panel component specifically without caring about the signal-detail component state.

    Candidate 3: E2E Test Fixture — Explicit Alpine Ready Signal

    Worth exploring

    Files

    views/layout.ejs, tests/e2e/signal-chat-overlay.spec.ts

    Problem — Shallow Test Wait Strategy

    Current tests wait for window.Alpine existence plus computed style checks. This is a shallow interface: the test knows too much about Alpine internals (x-show, display:none) rather than waiting for a high-level "ready" signal. The deletion test shows that removing these inline waits and replacing with one central ready signal concentrates the Alpine knowledge in the layout module.

    Solution

    1. Add window.__alpineComponentsReady = false to the layout.
    2. After Alpine loads, use Alpine.directive() or a MutationObserver to detect when all x-data components are initialized, then set the flag.
    3. Simplify all 5 tests to: await page.waitForFunction(() => window.__alpineComponentsReady).

    Benefits

    • Locality: Alpine initialization knowledge moves from 5 test files to 1 layout file.
    • Leverage: One ready signal serves all E2E tests, not just chat overlay tests.

    Top Recommendation

    Tackle Candidate 1 + Candidate 2 together. They are complementary fixes that address the problem at two levels:

    • Candidate 1 (Pin version) eliminates environment variance — the same Alpine code runs everywhere. This is a 2-line change with immediate impact.
    • Candidate 2 (Unnest chatPanel) eliminates the architectural coupling that makes nested initialization fragile. This is a structural fix preventing future regressions.

    Together they create a deep seam: one pinned version at the layout boundary, one independent component per page feature. The deletion test passes for both — removing either one causes complexity to reappear across multiple callers (tests, views).
