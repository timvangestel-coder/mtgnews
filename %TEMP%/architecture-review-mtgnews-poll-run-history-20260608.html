Architecture Review: Poll Run History Deepening

Generated 2026-06-08 · improve-codebase-architecture skill

Context

The admin [Polling] tab presents rich, state-aware Poll Run progress via a deep view model (RunState) produced by PollRunManager.runState(). It maps raw DB statuses to UI-friendly enums, computes per-channel step progress (X/Y done/total), and drives a 5-branch colour-coded display for running, aborted, and completed runs.

The Run History pages (/polls list + /polls/:id-detail) present the same underlying data through a completely different, shallow path: raw DB rows passed directly to templates with inline status logic. The result is divergent terminology, missing progress granularity (no X/Y per channel in detail), and duplicated display logic scattered across EJS templates.

Friction: two independent presentation paths for the same domain concept (Poll Run state), one deep, one shallow. Every new status or display rule must be implemented twice, increasing the bug surface and reducing locality.

Candidate 1: Unify Run History on RunState

Strong
Files involved:
src/services/poll-query-service.ts
src/routes/polls-router.ts
views/polls.ejs
views/poll-detail.ejs
views/admin/_pollProgress.ejs
Problem:
PollQueryService is a shallow pass-through: its interface (listRuns(), getRunDetail()) exposes raw DB shapes (PollRunRow, PollRunProgressRow) directly to templates. All status mapping and display logic leaks into EJS, duplicating what PollRunManager.runState() already does for the admin tab. The Run History detail page shows per-channel signals_found but not signalsDone/signals_found progress (X/Y), making it impossible to see per-channel completion from the history view.
Solution:

Promote RunState (and its constituent PollRunStep) from PollRunManager's private domain to a shared view model consumed by both the admin [Polling] tab and the Run History pages. Concretely:

  • Move RunState + PollRunStep + status mapping into PollQueryService as a public method getRunState(runId): RunState | null. This is the seam: the query service becomes the single source of truth for Poll Run view models.
  • Extract status-mapping functions (mapStatus(), mapStepStatus()) from PollRunManager into a pure module (e.g., src/utils/poll-status-map.ts) so both the Manager and the Query Service share the same mapping logic.
  • Update PollQueryService.getRunDetail() to return RunState (or a superset including it) instead of raw progress rows. The list view (ListRunsResult) stays as-is since it needs aggregate columns, but the detail view gets the deepened model.
  • Replace inline EJS logic in poll-detail.ejs with the same 5-branch step display used by _pollProgress.ejs, driven by the pre-computed PollRunStep status enum.
  • Refactor PollRunManager.runState() to call the shared mapping functions instead of duplicating them. The Manager still builds RunState, but delegates mapping to the pure utility — two adapters at the same seam.
Benefits:
  • Locality: Status mapping logic lives in one place (poll-status-map.ts). Adding a new DB status or changing display rules touches a single module, not two templates and two classes.
  • Leverage: The PollQueryService.getRunState() interface is small (one method) but returns rich, pre-computed data. Callers get mapped statuses, step progress (X/Y), and display names without knowing about DB schemas.
  • Testability: Pure mapping functions are trivially unit-testable. The Query Service method can be tested with in-memory SQLite. Template logic shrinks to simple switch-on-enum, reducing the need for template-level tests.
  • Consistency: Admin tab and Run History detail show identical per-channel states (same colours, same labels, same X/Y format) because they consume the same view model.

Deletion Test

Imagine deleting PollQueryService.getRunDetail(). The complexity doesn't vanish — it reappears in the route handler (polls-router.ts) which would need to call DB functions directly, and in the template which would need inline status mapping. The module is currently a pass-through (shallow), but deepening it concentrates that complexity productively: one deep method instead of N shallow callers.

Candidate 2: Extract Step Display Logic into a View Helper

Worth exploring
Files involved:
views/admin/_pollProgress.ejs
views/poll-detail.ejs
Problem:
The 5-branch step display logic (fetching → blue, failed → red, none → grey, X/Y processing → blue, Y/Y done → green) is copy-pasted as inline EJS JavaScript across _pollProgress.ejs in three places (running, aborted, complete states). That's 3 × 5-branch blocks of near-identical logic. Any change to colours or labels requires editing 3 template blocks.
Solution:

Extract the step-label + step-color computation into an Express view helper (e.g., res.locals.stepLabel(), res.locals.stepColor()) that takes a PollRunStep and run status, returning the display string and CSS class. The template then calls <%= stepLabel(step, state.status) %> instead of inline if/else blocks.

Benefits:
  • Locality: Display rules for one concept (channel step state) live in one function, not scattered across template branches.
  • Testability: Pure function of (step, runStatus) → {label, color}. Trivially testable with table-driven tests covering all 15 state combinations (3 run states × 5 step states).

Before — Triple-copy inline logic

if (step.status === 'fetching') {
  stepLabel = 'fetching';
  stepColor = 'text-blue-600';
} else if (step.status === 'failed') {
  ...
// repeated 3x in _pollProgress.ejs

After — Single helper call

<% const {label, color} = 
   stepDisplay(step, state.status); %>
<%= label %>

Candidate 3: Add HTMX Polling to Run History Detail for Active Runs

Speculative
Files involved:
views/poll-detail.ejs
src/routes/polls-router.ts
Problem:
The Run History detail page is static — it shows a snapshot. If a user opens the detail of a currently-running poll, they see stale data with no auto-refresh. The admin [Polling] tab gets live updates via HTMX polling (hx-trigger="every 3s"), but the Run History pages don't.
Solution:

When rendering poll-detail for a run with status running, include HTMX polling attributes on the channel breakdown table that call back to a lightweight endpoint (e.g., GET /polls/:id/progress) returning just the table fragment. Stop polling when status changes to done/failed/aborted.

Benefits:
  • Consistency: Both admin and history pages show live progress for active runs.
  • Leverage: The same RunState view model powers both the full-page detail and the HTMX fragment swap.

This candidate depends on Candidate 1 (unified RunState) landing first. Skipping it is fine — it's a UX nicety, not an architectural fix.

Top Recommendation

Tackle Candidate 1 first: Unify Run History on RunState. It's the foundational change that eliminates the dual-path problem. Once the Query Service exposes a deep view model, both templates become thin renderers — and Candidates 2 (view helper) and 3 (HTMX polling) become incremental improvements on top of the unified foundation.

The sequence: Candidate 1 → Candidate 2 (inline, low effort) → Candidate 3 (optional UX polish). Total estimated impact: high leverage, moderate effort, immediate consistency gain.
