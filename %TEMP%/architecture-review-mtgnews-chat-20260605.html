Split LLM Module — Streaming Seam

Strong

src/llm.ts · src/http-retry.ts

Before — shallow

                flowchart LR
                  A[Route Handler] --> B[analyzeSignal()]
                  B --> C[callLlmWithRetry()]
                  C --> D[fetchWithRetry()]
                  D --> E[Response.json()]
                  E --> F["DB write (inside llm.ts)"]
                  classDef leak stroke:#dc2626,stroke-width:2px,color:#dc2626;
                  class B,F leak

After — deepened

                flowchart LR
                  A[Route Handler] --> B["llm.ts (deep)"]
                  B --> C1[analyzeSignal()]
                  B --> C2[chatStream()]
                  C1 --> D[HttpSyncAdapter]
                  C2 --> E[HttpStreamAdapter]
                  D --> F[fetchWithRetry]
                  E --> G[fetchAsStream]
                  style B fill:#0f172a,color:#fff,stroke:#0f172a

Problem: llm.ts is shallow — the interface (analyzeSignal(db, videoId, config)) exposes implementation details (DB writes inside the LLM call). No seam for streaming. Chat feature would duplicate or monkey-patch this module.

Solution: Extract an HTTP adapter interface behind a seam: one sync adapter (fetchWithRetry → .json()) for analysis, one stream adapter (fetchAsStream → AsyncGenerator) for chat. DB writes move out of llm.ts into the caller (service layer).

  • ●Locality: LLM HTTP logic concentrates in one module — sync + stream variants side by side
  • ●Leverage: Two adapters at the seam = real seam. Tests can inject mock adapter without mocking fetch
  • ●Interface shrinks: callers receive typed results or streams — no DB coupling leaks through

Extend PromptAssembler — Chat Template Seam

Worth exploring

Before — single purpose

After — polymorphic

                flowchart LR
                  A[SignalContext] --> B["assemble()"]
                  C[ChatContext] --> D["assembleChat()"]
                  B --> E["{TRANSCRIPTION} + {FILTER_TEXT}"]
                  D --> F["{SUMMARY} + {HISTORY} + {QUESTION}"]
                  style B fill:#1e293b,color:#fff,stroke:#1e293b
                  style D fill:#1e293b,color:#fff,stroke:#1e293b

Problem: PromptAssembler is clean but single-purpose. Chat needs its own prompt template with different placeholders ({SUMMARY}, {HISTORY}, {QUESTION}). Adding a sibling function keeps the module deep — one interface for all prompt rendering.

Solution: Add assembleChat(chatContext) alongside assemble(signalContext). Both use the same three-tier resolution chain (topic override → DB default → compiled fallback). Chat template stored in app_settings.chat_prompt_template.

  • ●Locality: All prompt rendering in one module — no template logic scattered across route handlers
  • ●Leverage: Same three-tier resolution pattern reused for chat — one config mechanism, two prompt types

Deepen Service Layer — Chat Manager

Strong

src/services/signal-query-service.ts · src/routes/signals-router.ts

Before — thin wrapper

                flowchart TD
                  A[Route Handler] --> B["SignalQueryService.summarizeSignal()"]
                  B --> C["getLlmConfig()"]
                  B --> D["analyzeSignal(db, videoId, config)"]
                  style B fill:#f1f5f9,color:#64748b,stroke-dasharray: 4 4

After — Manager absorbs orchestration

                flowchart TD
                  A[Route Handler] --> B["ChatManager.ask()"]
                  B --> C["resolveSignalContext()"]
                  B --> D["getChatHistory()"]
                  B --> E["assemblePrompt()"]
                  B --> F["streamLlm() → persist"]
                  style B fill:#0f172a,color:#fff,stroke:#0f172a

Problem: Current service layer is shallow — SignalQueryService.summarizeSignal() is a 3-line pass-through calling analyzeSignal() directly. The route handler + service together hold the orchestration logic, but neither owns it fully.

Solution: Create ChatManager following the existing naming convention (-Manager suffix for write/CRUD). The manager absorbs full orchestration: resolve context → fetch history → assemble prompt → stream LLM → persist answer. Route handler becomes a thin 5-line delegate.

  • ●Deletion test: deleting the manager would scatter orchestration across route + LLM module — it earns its keep
  • ●Locality: chat bugs concentrate in one module instead of spanning route → service → LLM → DB
  • ●Leverage: single test surface: test the manager with in-memory SQLite + mock stream adapter

DB Init — Table Registration Seam

Speculative

src/db/init-db.ts

Before — monolithic init

                flowchart LR
                  A[init-db.ts] -->|"CREATE TABLE signals"| B[(SQLite)]
                  A -->|"CREATE TABLE channels"| B
                  A -->|"CREATE TABLE topics"| B
                  A -->|"CREATE TABLE ... 8 more"| B

After — registry pattern

                flowchart LR
                  A[init-db.ts] -->|"register()"| B["Schema Registry"]
                  C[signals schema] -.-> B
                  D[channels schema] -.-> B
                  E["chat schema (new)"] -.-> B
                  B -->|"CREATE all"| F[(SQLite)]

Problem: init-db.ts is a monolithic pass-through — every new table requires editing the same file. No seam for feature-scoped schema registration.

Solution: Introduce a simple registry: each domain module exports its DDL, init collects and runs all. Adding signal_chat becomes a one-file change in the chat module rather than editing the shared init.

  • ●Locality: chat schema lives next to chat code — not buried in a shared init file
  • ○Low priority for greenfield project with stable schema — defer until table count grows

Top Recommendation

Split LLM Module — Streaming Seam

This is the foundation for the chat feature. Without a streaming seam in the LLM module, every other piece (ChatManager, PromptAssembler extension) has to work around the sync-only constraint. Deepening this module first creates the adapters that everything else plugs into — it's the highest-leverage change.
