Delete Script: Missing signal_chat FK Cleanup

Strong

Files

scripts/delete-by-age.mjs
src/db/init-db.ts

Problem

The script deletes entity_mentions before signals, but omits the signal_chat table. Since signal_chat.signal_video_id REFERENCES signals(video_id) and foreign keys are enabled (PRAGMA foreign_keys = ON), deleting a Signal that has Q&A entries will throw a foreign key constraint violation — the deletion fails partway through, leaving orphaned entity_mentions deleted but signals intact. The module is shallow: its interface (parse args → run SQL) exposes nearly all implementation detail, and the deletion order logic is scattered across inline statements with no locality.

Before / After — Deletion Order Sequence

BEFORE — Shallow (broken)

  1. SELECT signals to delete
  2. DELETE entity_mentions ✓
  3. DELETE signals → FK violation on signal_chat ✗
  4. (never reaches counter recalc)

AFTER — Deepened (correct)

  1. SELECT signals to delete
  2. DELETE signal_chat ✓ (FK parent of signals)
  3. DELETE entity_mentions ✓
  4. DELETE signals ✓ (children gone)
  5. Recalculate poll_run counters ✓

Solution

Insert a DELETE FROM signal_chat WHERE signal_video_id IN (...) step before deleting signals. The deletion order must respect the FK dependency: children (signal_chat, entity_mentions) before parent (signals). This adds one DELETE statement and concentrates the deletion-order knowledge in a single place (locality).

Benefits

  • Correctness: eliminates FK constraint crash when signals have Q&A entries
  • Locality: all child-table cleanup in one sequential block
  • Leverage: the script interface stays the same (same args, same output) while handling a wider set of valid inputs
  • Depth: the module now absorbs FK dependency logic internally rather than leaking it to callers who must know the deletion order

ADR conflict: None. This is a correctness fix aligned with the existing schema in init-db.ts.

Top Recommendation

Add signal_chat cleanup before signal deletion. This is a single-statement fix with zero architectural risk. The script currently handles one child table (entity_mentions) but misses the second one (signal_chat). Adding it increases depth — the module now correctly handles all FK dependencies behind the same small interface — and provides locality by keeping all deletion-order logic in one file.
