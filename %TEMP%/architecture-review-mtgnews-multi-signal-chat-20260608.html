Strong

                flowchart LR

                  class B,C,D shallow

After — deep scope resolver

                flowchart LR
                  A[chat-router] -->|scope: {videoId? topicKey? channelId?}| B[ChatManager]
                  B --> C[resolveScope(scope)]
                  C -->|single videoId| D1[querySignals: 1 row]
                  C -->|topicKey/channelId| D2[querySignals: N rows]
                  D1 -.-> E[(SQLite)]
                  D2 -.-> E
                  B --> F[assemblePrompt(signals[])]
                  classDef deepBox stroke:#0f172a,stroke-width:3px,fill:#1e293b,color:#fff;
                  class B,C deepBox

Problem: resolveSignalForChat() is hardcoded to a single video_id — the interface leaks the single-signal implementation into every caller. Adding multi-signal means either duplicating resolution logic or widening the ChatManager interface with a second set of methods.

Solution: Introduce resolveScope(scope) as an internal module that returns SignalContext[]. The scope object is polymorphic: single-signal mode passes {videoId}; list mode passes {topicKey, channelId}. ChatManager delegates to it instead of calling SQL directly.

Wins:

  • Locality: signal resolution in one module, not scattered across ChatManager + router
  • Leverage: same resolver serves both single and multi-signal modes
  • Interface shrinks: callers pass scope object instead of individual filter params
  • Tests hit one interface: mock resolveScope() for both paths

Strong

Before — TimestampFormatter handles everything

After — CitationFormatter at a new seam

                flowchart LR
                  A[LLM raw text] --> B[CitationFormatter.format(text, signalMap)]
                  B -->|"<>" → pill| C1[Signal citation links]
                  B -->|"T:ss" → [MM:SS]| C2[Timestamp pills]
                  C1 --> D[persisted answer]
                  C2 --> D
                  classDef deepBox stroke:#0f172a,stroke-width:3px,fill:#1e293b,color:#fff;
                  class B deepBox

Problem: TimestampFormatter is a single-signal module — it converts timestamps to anchors within one signal's transcript context. Multi-signal answers need cross-signal citations (<>) that link to different signals entirely. Adding this logic to TimestampFormatter would shallow it further by mixing two concerns.

Solution: Create a new CitationFormatter module with format(text, signalMap). It handles both citation delimiters AND delegates timestamp formatting internally. Takes a signalMap: Record so it can render pill labels like "Signal Title · 02:00". TimestampFormatter becomes an internal dependency.

Wins:

  • Locality: all post-processing transforms in one module
  • Deep interface: two inputs (text + signalMap), one output — high leverage
  • TimestampFormatter becomes an internal adapter, not a public seam
  • Pure function: easy to test with string → string assertions
⚠ Contradicts CONTEXT.md: Current glossary says TimestampFormatter is "the transform seam in ChatManager." This deepens that seam into CitationFormatter, making TimestampFormatter internal. The ADR-equivalent decision in CONTEXT.md would need updating.

Problem: Three parallel assemble* functions with duplicated template resolution and placeholder replacement logic. Each new mode adds another function — shallow proliferation.

Solution: Unify behind a single assemble(request) interface where the request type encodes mode: analysis, chat (single), or multi-signal chat. Template resolution and placeholder injection become internal implementation details.

Wins:

  • Delete 2 shallow wrapper functions
  • Locality: template resolution in one place
  • Leverage: callers don't choose between assemble/assembleChat/assembleMultiSignalChat
