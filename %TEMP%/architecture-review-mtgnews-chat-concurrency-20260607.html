Problem

The ChatManager module is shallow — its interface (ask(videoId, question)) exposes the full implementation detail of direct LLM streaming with no queuing. Two simultaneous chat requests bypass all concurrency controls, hitting LM Studio raw. The Poll system uses a ConcurrencyPool (limit 3) for LLM analysis, but Chat routes around it entirely.

Solution

Introduce a ChatQueue module that sits between the chat-router and ChatManager. The queue reuses the ConcurrencyPool pattern: incoming requests are enqueued, processed with at most LLM_CONCURRENCY parallel slots, and SSE streams fan out to waiting browsers. The router interface stays thin — it just connects the SSE response to the queue's output stream.

Wins

  • ▸ locality: concurrency bugs concentrate in one module, not scattered across callers
  • ▸ leverage: same ConcurrencyPool pattern serves Poll + Chat — single test surface
  • ▸ parity: chat questions behave like poll signals (queued, rate-limited, persisted reliably)
  • ▸ enables pending state: queue naturally produces "asked but not answered" lifecycle

Problem

The signal_chat table requires both question AND answer (NOT NULL), so a row only exists after successful streaming. This makes it impossible to track the "asked but not yet answered" state needed for an X/Y progress ratio in the Signal List.

Solution

Allow answer TEXT (nullable). On question submit, INSERT immediately with answer = NULL. On stream success, UPDATE with the answer. On failure, optionally mark status. The ratio query counts COUNT(answer IS NOT NULL) / COUNT(*).

Wins

  • ▸ locality: pending/completed state lives in one table, queryable from Signal List
  • ▸ enables X/Y ratio where X can genuinely differ from Y during streaming
  • ▸ failed questions leave trace (answer=NULL + status='failed') instead of vanishing
