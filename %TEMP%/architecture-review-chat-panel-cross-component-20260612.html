Problem

TimestampNav and ChatPanel live in separate Alpine x-data scopes with no mechanism for one to signal the other. The onChatClick callback navigates the main page (split view + scroll) but cannot close the chat panel because ChatPanel's chatOpen state is behind a different Alpine boundary. This is a shallow coupling — the callback interface carries only navigation logic, leaving panel lifecycle uncoordinated.

Solution

Add a custom DOM event as the seam between the two Alpine scopes. The onChatClick callback dispatches chat-timestamp-clicked after navigation. ChatPanel listens in its init() and calls toggleChat() to close. Zero coupling: neither module knows about the other's internals.

Wins

  • Locality: close behaviour lives in one callback, not scattered across templates
  • Leverage: same event pattern reuses for any future cross-panel action (e.g., auto-open on new answer)
  • Interface shrinks: TimestampNav interface gains no new params — event is fire-and-forget
  • No ADR conflict: follows ScopeSource pattern of framework-agnostic JS modules communicating via DOM

Problem

TimestampNav is shallow — its interface (init({samePage, onSummaryClick, onTranscriptClick, onChatClick})) passes click events through to caller-provided callbacks without any lifecycle awareness. The module knows the click came from inside a chat panel but has no opinion on what happens to that panel afterward.

Solution

Add an optional closeSourcePanel?: boolean flag to TimestampNav's init options. When true and the click originates from inside a chat panel, TimestampNav queries for the ChatPanel's Alpine instance via DOM selector and sets chatOpen = false after firing the callback. This deepens the module by absorbing lifecycle logic that currently leaks into every caller.

Wins

  • Locality: panel-close logic concentrates in TimestampNav, not duplicated per-page
  • Leverage: multi-signal chat page gets same behaviour automatically (one adapter)

Coupling risk: This approach creates a direct DOM dependency from TimestampNav into ChatPanel's Alpine state. Contradicts the ScopeSource pattern of framework-agnostic modules. The event-based approach (Candidate 1) avoids this coupling.

Top recommendation

Candidate 1 — Event-based seam. Minimal change, zero coupling, follows the existing ScopeSource/TimestampNav pattern of framework-agnostic JS modules. Two lines in ChatPanel's init() to listen for the event; one line in onChatClick to dispatch it. The DOM event bus is the seam — both modules satisfy it independently without knowing each other's interfaces.
