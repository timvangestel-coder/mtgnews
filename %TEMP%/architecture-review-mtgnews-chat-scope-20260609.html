Interface mass — after (deep)

ScopeSource.fromURL()

One interface. All 5 callers cross one seam. Implementation absorbs URL parsing + normalization.

Problem

Scope reading scattered across 5 methods in chat-panel.js. Each re-implements URL parsing, normalization, and state sync. Alpine reactive properties drift from actual browser URL because updates only fire when chat is open.

Solution

Extract a pure ScopeSource.fromURL() module that reads window.location.searchParams and returns a normalized scope object. All callers use it at point-of-use instead of maintaining stale Alpine state. Delete _syncScopeFromUrl, savedTopicKey, savedChannelId.

Wins

  • Locality: scope bugs in one module
  • Leverage: 5 callers, 1 interface
  • Delete drift-prone Alpine state props
  • Pure function — trivially testable

4. Disambiguate empty-string topicKey in ChatScope

Replace ambiguous topicKey: '' with explicit scope type discrimination

Worth exploring
src/signal-chat-scope.ts src/routes/chat-router.ts src/services/chat-manager.ts

Before — 4 thin layers, each interpreting empty string differently (shallow)

Frontend: topicKey='' means "no filter selected"
Router: topicKey='' means "all signals" (list-scope)
Manager: topicKey='' stores as empty string in DB
History: topicKey='' matches WHERE topic_key = ''

Same value, 4 different meanings. Impossible to distinguish "user sent no filter" from "user selected all signals".

After — 1 deep module, explicit discrimination

ChatScope module — explicit type tag {scopeType: 'perSignal', videoId} | {scopeType: 'listAll'} | {scopeType: 'listFiltered', topicKey?, channelId?}

One interface. No empty-string ambiguity. Discriminated union enforces exhaustiveness at compile time.

Problem

topicKey: '' means different things at each layer. Frontend sends it for "no filter". Router interprets it as "all signals list-scoped". Manager stores empty string in DB. History query matches on empty string. Fundamental ambiguity makes correct filtering impossible.

Solution

Replace flat ChatScope with a discriminated union tagged by scope type. TypeScript enforces exhaustive handling. Empty string disappears as a valid topicKey value — replaced by explicit scopeType: 'listAll'.

Wins

  • Locality: ambiguity resolved in one module
  • Compile-time exhaustiveness checks
  • Eliminates empty-string confusion
  • Cleaner DB queries (NULL vs '' is clear)

⚠ Contradicts issue #135 design decision

Issue #135 explicitly chose empty-string topicKey as "all signals" indicator. This candidate proposes reversing that decision because the ambiguity caused real bugs (chatissues.md Issues 1 & 2). Worth reopening because the original design cannot distinguish "no filter selected" from "intentionally all signals".

  • Router shrinks, manager deepens

Top Recommendation

Candidate 1 + 2 + 3 — tackle together as a single architectural improvement sprint.

The three chatissues.md bugs all trace to the same root cause: scope data has no single source of truth between frontend and backend. Candidate 1 creates the ScopeSource seam on the frontend (eliminates drift). Candidate 2 replaces fragile regex tests with real E2E behavior tests. Candidate 3 adds integration coverage for the full POST→DB→history path.

Together they create a testable architecture that prevents these bugs from recurring: one deep module reads scope, E2E tests verify it works in the browser, and integration tests catch corruption at the backend seam. Candidates 4 and 5 are follow-ups that can wait until after the core issue is resolved.
