Architecture Review: Chat Delete + Cancel Mechanism

2026-06-11 · mtgnews · One deepening opportunity

Problem Statement

When a user deletes a chat question while it's still being processed, the DELETE route removes the DB row but the LLM work continues running in the background. The ConcurrencyPool task runs to completion, the HTTP request to LM Studio finishes, and the answer is silently discarded because the target DB row no longer exists.

The project already has a fully-working abort pattern in PollRunManager: per-run AbortController, signal threading through worker → analyzeSignal() → callLlmSync() → fetchWithRetry(). The LLM layer (llm.ts) already accepts an optional abortSignal parameter.

The chat processing chain uses none of this infrastructure. There is no per-task AbortController, no task registry, and no signal threading from ChatQueue → ChatManager.process() → callLlmSync().

Candidate: ChatTaskController — per-question abort seam

Strong

Files

  • src/chat-queue.ts
  • src/services/chat-manager.ts
  • src/llm.ts (already supports abortSignal)

Seam

ChatQueue is the seam. It sits between the HTTP route layer and ChatManager.process(). Adding a per-task AbortController registry here creates a real cancel seam without touching ConcurrencyPool or the LLM layer.

Problem

Three independent systems with no reverse signal:

ModuleOn deleteMissing
ChatManager.delete()Removes DB row onlyDoes not signal the running task
ChatQueue._dispatchProcess()Nothing — fire-and-forgetNo AbortController, no cancel hook
ConcurrencyPool.run()Nothing — runs to completionNo per-task abort (by design)

Solution

Add a Map registry to ChatQueue. On enqueue, create an AbortController per task and thread the signal through process() → callLlmSync(). On delete, fire controller.abort() before removing the DB row.

  1. ChatQueue gains _controllers = new Map
  2. _dispatchProcess(id) creates a controller, stores it, passes signal to process()
  3. ChatManager.process(id, signal?) threads the signal into callLlmSync(config, prompt, { abortSignal: signal })
  4. delete(id) on ChatQueue first checks for an active controller → fires abort() → then delegates to chatManager.delete(id)
  5. Task catches AbortError gracefully (same pattern as poll worker: silent return on abort path)

Benefits

  • Locality: All cancel logic concentrated in ChatQueue. No changes to ConcurrencyPool (which correctly stays consumer-agnostic).
  • Leverage: Reuses the existing abortSignal seam in llm.ts — no new HTTP infrastructure needed.
  • Testability: ChatManager.process() now accepts an optional AbortSignal, making it testable with mock signals. ChatQueue.cancel() is a pure in-memory operation.
  • Resource savings: Aborts the LLM HTTP request immediately instead of waiting for full generation. Saves tokens, time, and network bandwidth.

Reference: PollRunManager abort pattern

The same abort pattern already works for poll runs. The chat cancel mechanism mirrors it exactly:

PollRunManager (existing)

  • activeRuns: Map
  • startRun() → new AbortController()
  • workerProcessRun(runId, {signal})
  • analyzeSignal(..., signal)
  • callLlmSync(..., {abortSignal: signal})
  • abortPollRun() → controller.abort()

ChatQueue (proposed)

  • _controllers: Map
  • enqueue() → new AbortController()
  • _dispatchProcess(id) → process(id, signal)
  • process(id, {abortSignal})
  • callLlmSync(..., {abortSignal: signal})
  • cancel(id) → controller.abort()

Top Recommendation

Add a cancel seam to ChatQueue. This is the single smallest change with the highest leverage: one Map, one method (cancel(id)), and signal threading through two existing methods. The LLM layer already supports abortSignal. The poll abort pattern provides a proven reference implementation.

Estimated scope: ~3 files touched, ~40 lines net change. No breaking API changes to ConcurrencyPool or llm.ts.
